## 영상 핵심 요약

1. `조건검색`은 기간/유형 조회만으로 찾기 어려운 전표를 거래처, 계정, 금액, 세액, 전자상태 등 세부 조건으로 좁혀 찾는 기능이다.
2. `액션바`는 코드변환, 일괄적용, 일괄분개, 전자발행, 전표복사/이동, 인쇄, 삭제, 체크펜처럼 현재 선택 전표에 즉시 영향을 주는 작업을 모아 둔 영역이다.
3. `툴바 기능`은 자동전표검증, 행/열 고정, 데이터정렬, 중복전표검색, 계산기처럼 조회/검증/작업 보조 기능을 제공한다.
4. `안내메시지 영역`은 단축키, 일정관리, 작성자, 메모처럼 화면 하단에서 사용자가 놓치기 쉬운 보조 정보를 제공한다.
5. 대량 변경 기능은 실무 시간을 줄여주지만, 마감기간, 전자발행 상태, 외부전송 전표, 자동수집 전표에는 제한을 두어야 한다.

## 기능별 상세 정리

### 1. 조건검색

조건검색은 특정 기간 전체를 훑는 대신, 원하는 조건을 조합해 전표를 찾는 기능이다. 실무에서는 다음 상황에서 많이 쓴다.

- 특정 거래처의 매출/매입 전표만 찾을 때
- 특정 과세유형, 예를 들어 `11.과세`, `54.불공`, `57.카과`만 확인할 때
- 공급가액, 부가세, 공급대가가 특정 금액인 전표를 찾을 때
- 전자세금계산서 입력/발행/국세청 전송 상태를 확인할 때
- 자동수집자료 전송 후 확정되지 않은 전표를 찾을 때
- 특정 계정과목이나 관리항목이 들어간 분개를 찾을 때
- 신고 전 중복, 누락, 사업자번호 오류 가능성이 있는 전표를 추려낼 때

사용 흐름:

1. 기본 조회기간을 먼저 정한다.
2. 매출/매입 또는 전체 범위를 선택한다.
3. 조건검색을 열어 세부 조건을 입력한다.
4. 조회 결과가 원하는 범위인지 확인한다.
5. 결과 전표에 대해 검토, 일괄적용, 인쇄, 중복검색 등 후속 작업을 실행한다.

주의할 점:

- 조건검색 결과가 곧 처리 대상이 되므로, 일괄변경 전에 조회 조건과 선택 건수를 확인해야 한다.
- 신고기간을跨월로 잡으면 예정/확정 신고기간이 섞일 수 있다.
- 거래처코드 없이 입력된 전표는 세금계산서합계표 등에 반영되지 않을 수 있으므로, 검색 결과에서 거래처 누락 여부를 함께 본다.

## 액션바 기능

액션바는 선택 전표에 직접 변화를 주는 기능이 많다. 조회 기능이 아니라 데이터 변경 기능으로 보고 접근해야 한다.

### 1. 코드변환

코드변환은 이미 입력된 전표의 코드값을 다른 코드로 바꾸는 기능이다. 거래처 통합, 계정과목 정정, 관리항목 정정처럼 같은 오류가 여러 건에 반복되어 있을 때 사용한다.

대표 사용 예:

- 거래처가 잘못 선택된 전표를 올바른 거래처로 일괄 변경
- 임시 계정으로 입력된 전표를 실제 비용/자산 계정으로 변경
- 프로젝트, 현장, 부서, 사원 코드가 잘못 들어간 전표를 정정
- 사업자등록번호가 같은 중복 거래처를 하나로 정리

처리 전 체크:

- 변환 전 코드와 변환 후 코드가 같은 마스터 유형인지 확인한다.
- 세금계산서합계표/계산서합계표의 집계 키가 거래처코드와 사업자등록번호에 의존하므로, 거래처 변환은 신고서식 대사까지 포함한다.
- 하단 계정과목 변경은 원장, 재무제표, 고정자산매입/일반매입 분류에 영향을 준다.
- 마감된 기간은 직접 변환하지 않고 마감해제 또는 조정 절차를 거친다.

BK 설계 포인트:

- 코드변환은 `bulk_update`가 아니라 감사 가능한 `bulk_correction` 명령으로 설계해야 한다.
- 전/후 코드, 대상 기간, 대상 건수, 실행자, 실행 사유를 저장한다.
- 신고마감, 전자발행, 외부전송 상태가 있는 전표는 변환 가능 여부를 정책으로 분기한다.

### 2. 일괄적용

일괄적용은 여러 행에 같은 값을 넣거나 바꾸는 기능이다. 반복 입력을 줄이는 용도이며, 값이 동일해야 하는 범위가 명확할 때만 사용한다.

사용 가능성이 높은 값:

- 거래처명 또는 거래처 코드
- 적요
- 관리항목: 사원, 부서, 현장, 프로젝트
- 주류 여부 등 부가 관리값
- 특정 보조 입력값

실무 팁:

- 조건검색으로 범위를 좁힌 뒤 일괄적용하면 실수를 줄일 수 있다.
- 일괄적용 후에는 표본 몇 건이 아니라 전체 건수와 합계 금액이 의도대로 남았는지 확인한다.
- 세무 신고에 영향을 주는 값, 예를 들어 거래처와 과세유형은 일괄적용 후 부속서류 반영까지 확인한다.

### 3. 일괄분개

일괄분개는 여러 전표의 하단 분개를 한 번에 생성하거나 보정하는 기능이다. 상단 세무자료는 이미 맞는데 하단 계정과목/차변/대변이 누락되었거나 반복되는 경우에 유용하다.

사용 시나리오:

- 동일한 매출 유형 전표에 같은 매출 계정과 채권 계정을 적용
- 동일한 매입처의 반복 비용 전표에 비용 계정을 적용
- 카드매입 전표에 카드채무/미지급금 계정을 일괄 적용
- 분개유형 선택 후 하단 분개가 비어 있는 전표를 보완

주의할 점:

- 상단 세무자료와 하단 분개 합계가 다르면 부가세신고서의 과표/수입금액제외 등 차액 항목에 영향을 줄 수 있다.
- `2.외상` 분개유형은 외상매출금/외상매입금 계정이 자동 적용되고 계정 수정이 제한될 수 있으므로, 계정 변경이 필요하면 `3.혼합`을 검토한다.
- 고정자산매입, 불공제 매입, 카드/현금영수증 수취분은 하단 계정에 따라 신고서 반영 위치가 달라질 수 있다.

### 4. 전자발행

전자발행은 선택한 매출 세금계산서/계산서 전표를 전자세금계산서 또는 전자계산서로 발행하는 기능이다.

대상:

- `11.과세` 매출 세금계산서
- `12.영세` 영세율 세금계산서
- `13.면세` 계산서
- 일부 역발행/매입 발행 흐름은 별도 정책과 상태값을 따른다.

처리 전 체크:

- 거래처 담당자 이메일이 등록되어 있는지 확인한다.
- 회사 정보, 공급자 정보, 거래처 사업자등록번호가 정확한지 확인한다.
- 품명, 수량, 단가, 공급가액, 부가세가 증빙 기준과 일치하는지 확인한다.
- 이미 전자입력 또는 전자발행 상태인 전표를 중복 발행하지 않는지 확인한다.

처리 후 체크:

- WEHAGO 처리상태를 확인한다.
- 국세청 전송결과를 확인한다.
- 전송실패, 반송, 미전송 상태를 별도 검토한다.
- 합계표에서 전자/전자외 구분이 의도대로 반영되는지 확인한다.

BK 설계 포인트:

- 전자발행은 외부 효과이므로 DB 트랜잭션 안에서 직접 호출하지 않고 outbox/worker로 분리한다.
- 발행 요청, 발행 성공, 국세청 전송 성공, 실패, 취소를 상태머신으로 관리한다.
- 재시도는 멱등키를 기준으로 수행해 중복 발행을 막는다.

### 5. 전표복사와 전표이동

전표복사는 기존 전표를 다른 일자로 복제하는 기능이고, 전표이동은 기존 전표의 일자를 옮기는 기능이다.

전표복사 사용 예:

- 매월 반복되는 동일 거래를 다음 달로 복사
- 같은 거래처와 같은 품목의 반복 매출을 빠르게 생성
- 전년도/전월 입력 패턴을 참고해 신규 전표 입력

전표이동 사용 예:

- 거래일자 오입력 정정
- 월말/월초 귀속 기간 정정
- 증빙일자와 전표일자 불일치 정정

주의할 점:

- 복사는 새 전표를 만들고, 이동은 기존 전표의 귀속일자를 바꾼다는 점이 다르다.
- 일자를 바꾸면 부가세 신고기간, 합계표 집계기간, 원장 기간조회가 달라진다.
- 전자발행, 전송결과, 자동수집 원천, 자금관리, 차량관리, 일마감 상태가 있는 전표는 복사/이동 제한이 필요하다.
- 이동 전후 전표번호 채번 정책이 유지되어야 한다.

### 6. 인쇄

인쇄는 입력된 전표 또는 세금계산서 양식의 출력물을 생성한다.

확인할 항목:

- 출력 대상 기간과 선택 전표
- 영수/청구 표시
- 공급자/공급받는자 정보
- 품목이 여러 개인 경우 인쇄 품목 수와 `~외` 표시
- 전자세금계산서인지 종이 세금계산서인지
- 사업자단위과세, 책번호/일련번호, 외상거래 표시 등 선택 조건

BK 설계 포인트:

- 인쇄물은 단순 화면 출력이 아니라 신고/증빙 산출물이 될 수 있으므로 생성 이력을 남긴다.
- PDF/인쇄 재생성 시점과 원본 전표 상태를 함께 보관한다.

### 7. 삭제

삭제는 가장 위험도가 높은 액션바 기능이다. 매입매출전표는 상단 세무자료와 하단 분개자료가 동시에 연결되어 있으므로, 삭제는 신고서식과 원장을 모두 바꿀 수 있다.

삭제 전 체크:

- 전표가 마감기간에 속하는지 확인한다.
- 전자발행 또는 국세청 전송이 완료된 전표인지 확인한다.
- 자동수집자료에서 전송된 전표인지 확인한다.
- 삭제 후 원천 자료 상태가 어떻게 바뀌는지 확인한다.
- 신고서 마감 또는 전자신고 파일 생성 이후인지 확인한다.

BK 설계 포인트:

- 실제 물리 삭제보다 취소/삭제 상태 전이를 우선한다.
- 삭제자, 삭제일시, 삭제 사유, 원전표 식별자를 감사로그에 남긴다.
- 신고/마감 이후 삭제는 마감해제 또는 수정신고 워크플로로 우회한다.

### 8. 체크펜

체크펜은 전표를 눈에 띄게 표시하는 검토 보조 기능이다. 최종 회계처리를 바꾸는 기능은 아니지만, 협업 검토에서는 중요한 신호가 된다.

활용 예:

- 거래처 확인 필요
- 증빙 원본 확인 필요
- 과세유형 검토 필요
- 분개 검토 완료
- 신고 전 재확인 대상

BK 설계 포인트:

- 체크펜은 개인 표시인지 조직 공유 표시인지 정책을 정해야 한다.
- 원장 표시 여부가 환경설정과 연결되면, 표시 범위와 권한을 분리해야 한다.
- 체크펜을 감사로그와 동일시해서는 안 된다. 감사로그는 별도 시스템 기록으로 남겨야 한다.

## 툴바 기능

### 1. 자동전표검증

자동전표검증은 자동전표처리 메뉴에서 전송되었거나 전송 예정인 자료 중 확정 상태와 입력 정합성을 점검하는 기능이다.

검증 관점:

- 전자세금계산서/전자계산서 수집자료와 매입매출전표의 일치 여부
- 사업자등록번호 일치 여부
- 과세유형 적합성
- 면세사업자 매출유형 오류
- 자동전표 전송 후 미확정 상태
- 상단 세무자료와 하단 분개자료의 금액 차이
- 중복 입력 가능성

처리 후:

- 검증 오류는 자동수정하지 않고 사용자가 확인해야 한다.
- 신고자료에 반영하기 전 확정/제외/수정 상태를 명확히 한다.
- 미확정 전표를 신고서 불러오기 전에 정리한다.

### 2. 행고정, 열고정

행고정과 열고정은 대량 전표 그리드에서 기준 정보를 계속 보이게 하는 화면 기능이다.

유용한 경우:

- 거래처와 금액을 보면서 하단 관리항목을 검토할 때
- 많은 컬럼이 있는 상태에서 과세유형, 공급가액, 부가세를 비교할 때
- 여러 전표를 일괄적용하기 전 선택범위를 눈으로 확인할 때

### 3. 데이터정렬

데이터정렬은 입력순, 날짜순, 거래처코드순, 사업자번호순 등으로 전표를 재배열하는 기능이다.

주의:

- 정렬은 화면 표시 순서를 바꾸는 기능으로 이해해야 한다.
- 전표번호 자체를 임의로 재채번하는 기능으로 사용하면 안 된다.
- 정렬 후 선택 범위가 바뀌었는지 확인하고 일괄작업을 실행한다.

### 4. 중복전표검색

중복전표검색은 같은 거래가 두 번 입력되었거나 자동수집자료와 수동입력자료가 겹치는 경우를 찾는 기능이다.

비교 기준:

- 일자
- 유형
- 계정
- 거래처
- 공급가액
- 부가세
- 공급대가
- 국세청승인번호
- 사업자등록번호

실무 판단:

- 완전히 같은 금액과 거래처라도 분할 거래일 수 있으므로 삭제 전에 원본 증빙을 확인한다.
- 전자세금계산서와 카드/현금영수증이 같이 있는 거래는 증빙 중복인지 결제수단 보조정보인지 구분한다.
- 중복 의심 상태는 확정 전 검토 대상으로 남기는 것이 안전하다.

### 5. 계산기

계산기는 공급대가, 공급가액, 부가세, 차액 등을 계산할 때 사용하는 보조 도구다.

사용 예:

- 공급대가에서 공급가액과 부가세를 분리
- 복수 품목의 합계 확인
- 단수 차이 계산
- 차변/대변 차액 확인

주의:

- 계산기 결과가 증빙 원본을 대체하지 않는다.
- 부가세 단수 차이는 증빙 금액과 신고서 금액을 모두 고려해 조정한다.

## 안내메시지 영역

### 1. 공통단축키

공통단축키는 반복 입력 속도를 높이는 도구다. 예를 들어 코드도움, 삭제, 인쇄, 복사, 이동, 조회 같은 기능을 키보드로 빠르게 실행할 수 있다.

운영 포인트:

- 단축키로 실행되는 기능도 버튼 클릭과 동일한 권한 검사를 받아야 한다.
- 삭제/이동/코드변환 같은 위험 작업은 단축키 실행에도 확인 절차를 둔다.

### 2. 일정관리

일정관리는 전표 입력 화면에서 관련 업무 일정을 확인하거나 등록하는 보조 기능이다.

활용 예:

- 부가세 신고 마감일
- 거래처 증빙 회신 예정일
- 전자세금계산서 발행/전송 확인일
- 내부 검토 완료 예정일

BK 설계 포인트:

- 일정은 전표 자체의 회계 데이터와 분리하되, 전표/거래처/신고기간과 연결 가능한 참조를 둔다.
- 담당자 배정과 알림은 운영 콘솔 또는 업무현황과 연계할 수 있다.

### 3. 작성자

작성자는 전표를 누가 만들었는지 확인하는 정보다. 협업 회계 시스템에서는 작성자만으로는 부족하고, 수정자, 검토자, 승인자, 마감자까지 추적해야 한다.

필요한 이력:

- 최초 작성자/작성일시
- 최종 수정자/수정일시
- 전자발행 요청자
- 삭제/복구/이동/코드변환 실행자
- 검토/승인/반려 이력

### 4. 메모

메모는 전표 처리 과정에서 남기는 보조 설명이다.

좋은 메모 예:

- 거래처 사업자번호 확인 요청
- 증빙 원본 미수취, 추후 보완 예정
- 공급가액 단수 차이 조정 사유
- 불공제 사유 검토 필요
- 전자발행 실패 후 재전송 예정

주의:

- 메모는 업무 참고용이며 법적 감사로그를 대체하지 않는다.
- 개인정보, 카드번호, 인증정보, 주민등록번호 등 민감정보를 메모에 남기지 않는다.

## 실무 체크리스트

### 일괄작업 전

- 조회 기간이 맞는지 확인한다.
- 조건검색 결과 건수와 합계 금액을 확인한다.
- 선택된 행만 처리되는지, 조회 전체가 처리되는지 확인한다.
- 마감기간 전표가 포함되어 있는지 확인한다.
- 전자발행/국세청 전송 완료 전표가 포함되어 있는지 확인한다.
- 자동수집자료에서 넘어온 전표인지 확인한다.

### 일괄작업 후

- 변경 건수와 변경 전/후 값을 확인한다.
- 매입매출장, 합계표, 부가가치세신고서 반영 결과를 확인한다.
- 계정별원장과 분개장을 확인한다.
- 중복전표검색으로 의도치 않은 중복을 확인한다.
- 변경 사유와 검토 결과를 메모 또는 감사로그에 남긴다.

### 삭제/이동/전자발행 전

- 해당 전표가 신고서 마감 후 자료인지 확인한다.
- 전자세금계산서 발행 또는 전송결과가 있는지 확인한다.
- 원천자료가 전자세금계산서/전자계산서/카드/현금영수증 자동수집자료인지 확인한다.
- 작업 권한이 있는지 확인한다.
- 처리 후 되돌릴 수 있는 절차가 있는지 확인한다.

## BK 설계 반영 포인트

1. 조건검색은 `JournalSearchQuery`로 정형화하고, 상단 세무헤더 조건과 하단 분개 조건을 분리해 검색해야 한다.
2. 코드변환, 일괄적용, 일괄분개는 대량 변경 명령으로 분리하고, 대상 전표 스냅샷과 변경 사유를 보관한다.
3. 전표복사와 전표이동은 전표번호 채번, 신고기간, 마감 잠금, 외부 원천 연결 상태를 검증한 뒤 실행한다.
4. 삭제는 물리 삭제가 아니라 취소/삭제 상태 전이로 설계하고, 원장/신고자료 재집계가 멱등하게 동작해야 한다.
5. 자동전표검증은 오류를 자동수정하지 않고 검토 상태와 오류 사유를 남긴다.
6. 중복전표검색은 전표 입력 화면뿐 아니라 전자세금계산서, 카드, 현금영수증 수집자료와도 비교해야 한다.
7. 전자발행은 outbox 기반 비동기 처리로 분리하고, 발행 요청과 국세청 전송결과를 상태머신으로 관리한다.
8. 체크펜과 메모는 협업 보조 기능으로 두되, 감사로그와는 별도 계층으로 둔다.
9. 작성자 정보는 작성자뿐 아니라 수정자, 검토자, 승인자, 삭제자, 전자발행 요청자까지 확장한다.
10. 단축키 실행도 UI 버튼 실행과 동일한 권한/검증/감사 정책을 적용한다.

## 구현 모델로 바꾸면 필요한 기능 단위

| 기능 단위 | 설명 |
|---|---|
| `JournalSearchQuery` | 기간, 유형, 거래처, 사업자번호, 계정, 금액, 전자상태, 관리항목 조건 검색 |
| `BulkCodeConversionCommand` | 코드변환 실행 단위, 전/후 코드와 대상 건수 기록 |
| `BulkApplyCommand` | 선택 전표의 동일 값 일괄 적용 |
| `BulkJournalizeCommand` | 하단 분개 일괄 생성/보정 |
| `ElectronicIssueCommand` | 전자세금계산서 발행 요청, outbox 및 멱등키 포함 |
| `VoucherCopyCommand` | 기존 전표 복사, 새 일자/번호 채번 |
| `VoucherMoveCommand` | 전표 귀속일자 이동, 신고기간/마감 검증 |
| `VoucherDeleteCommand` | 전표 삭제 상태 전이, 삭제 사유 필수 |
| `AutoVoucherValidation` | 자동전표 미확정/오류 상태 검증 |
| `DuplicateVoucherDetector` | 수동입력/자동수집/전자증빙 간 중복 후보 탐지 |
| `VoucherMarker` | 체크펜, 검토 표시, 사용자별/공유 표시 |
| `VoucherMemo` | 전표별 업무 메모, 민감정보 금지 정책 포함 |
| `VoucherAuditLog` | 변경, 삭제, 이동, 발행, 검증, 승인 이벤트의 불변 로그 |

## 후속 문서에서 이어갈 내용

3편은 전표 입력 화면 내부의 검색/액션/검증 도구를 다뤘다. 다음 편들을 계속 정리할 때는 아래 항목을 이어서 보면 좋다.

- 전자세금계산서/전자계산서 자동수집자료가 전표로 확정되는 흐름
- 자동전표검증 오류 유형과 해결 절차
- 매입매출전표와 부가가치세신고서 불러오기/마감의 연결
- 중복전표검색과 국세청자료검증의 차이
- 일괄 변경 기능을 BK 권한/감사로그/마감잠금 정책으로 모델링하는 방법
