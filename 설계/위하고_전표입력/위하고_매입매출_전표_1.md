## 핵심 요약

`매입매출전표입력`은 부가가치세 신고와 관련된 매출·매입 거래를 입력하는 메뉴다. 일반전표입력이 부가세 신고와 직접 연결되지 않는 일반 회계거래 중심이라면, 매입매출전표입력은 세금계산서, 계산서, 신용카드, 현금영수증, 수입세금계산서 등 신고자료와 연결되는 거래를 입력한다.

영상은 다음 흐름을 다룬다.

1. 전체탭과 유형별탭
2. 조회하기
3. 거래처 입력
4. 유형 입력
5. 유형별 선택방법
6. 품명 입력
7. 수량과 단가 입력
8. 전자 항목
9. 전자세금계산서 발행 방법
10. 분개 입력
11. 매출 세금계산서 발급 후 신용카드로 결제한 경우 입력방법
12. 관리항목 입력

핵심은 매입매출전표가 `상단부 세무자료`와 `하단부 회계분개`의 2층 구조라는 점이다. 상단부의 거래처, 과세유형, 공급가액, 부가세, 전자 여부는 부가세신고서, 세금계산서합계표, 계산서합계표, 매입매출장 같은 세무자료로 이어진다. 하단부 분개는 분개장, 계정별원장, 총계정원장, 재무제표로 이어진다.

## 매입매출전표의 기본 구조

| 구분 | 입력 위치 | 역할 | 후속 반영 |
|---|---|---|---|
| 상단부 | 일자, 거래처, 유형, 품명, 공급가액, 부가세, 전자, 분개, 관리 | 부가세 신고와 세금계산서·계산서·카드·현금영수증 관련 세무자료의 원천 | 부가세신고서, 세금계산서합계표, 계산서합계표, 매입매출장, 신용카드매출전표발행현황 |
| 하단부 | 구분, 계정과목, 차변/대변 금액, 거래처, 적요, 관리항목 | 회계 장부와 재무제표에 반영되는 분개 | 분개장, 계정별원장, 총계정원장, 재무제표 |

입력자는 상단부와 하단부가 서로 다른 목적을 가진다는 점을 항상 기억해야 한다. 상단부가 맞아도 하단부 분개가 틀리면 장부가 틀어지고, 하단부가 맞아도 상단부 과세유형이나 전자 여부가 틀리면 부가세 신고자료가 틀어진다.

## 전체탭과 유형별탭

| 탭 | 의미 | 사용 상황 | 주의점 |
|---|---|---|---|
| 전체탭 | 매출·매입 과세유형을 전체적으로 조회하고 입력 | 다양한 유형의 거래를 한 번에 확인하거나 수정할 때 | 거래유형이 섞여 있으므로 색상, 코드, 증빙을 함께 확인 |
| 유형별탭 | 특정 과세유형 중심으로 입력 화면을 좁혀 사용 | 반복적으로 같은 유형을 입력할 때 | 빠른 부가세 입력설정에서 표시할 유형을 정리해야 함 |

유형별 탭은 반복 입력을 빠르게 하지만, 잘못된 유형 탭에서 입력하면 신고자료 반영 위치가 달라질 수 있다. 예를 들어 계산서 매출을 과세 세금계산서 탭에서 입력하면 세금계산서합계표와 계산서합계표 반영이 뒤섞일 수 있다.

## 조회하기

조회는 기간 또는 월별 기준으로 수행한다. 매입매출전표는 부가세 과세기간과 직접 연결되므로 조회 기간이 특히 중요하다.

| 확인 항목 | 내용 |
|---|---|
| 기간/월별 | 특정 월 또는 기간 범위를 선택해 조회 |
| 거래일자 | 세금계산서 작성일, 공급시기, 카드승인일, 현금영수증 승인일 등과의 관계 확인 |
| 마감 상태 | 일마감 또는 부가세 마감 기간인지 확인 |
| 조회 조건 | 거래처, 유형, 금액, 전자 여부 등으로 필터링할 수 있음 |

실무에서는 입력 전에 조회를 먼저 실행해 같은 증빙이 이미 들어가 있는지 확인하는 것이 좋다. 자동수집 자료가 전송된 뒤 같은 증빙을 수기로 다시 입력하면 중복이 생길 수 있다.

## 거래처 입력

매입매출전표에서 거래처는 단순 명칭이 아니라 신고자료 집계 기준이다. 특히 세금계산서합계표와 계산서합계표는 거래처의 사업자등록번호 또는 주민등록번호를 기준으로 집계된다.

| 항목 | 설명 | 주의점 |
|---|---|---|
| 거래처코드 | 거래처등록에 등록된 상대방 코드 | 매입매출전표에서는 거래처코드 입력이 매우 중요 |
| 거래처명 | 거래처 상호 또는 성명 | 이름만 직접 입력하면 거래처별 신고 집계가 불안정해짐 |
| 사업자등록번호 | 합계표, 휴폐업 검증, 세금계산서 발행 기준 | 누락 또는 불일치 시 신고자료 오류 가능 |
| 조회 방법 | 더블클릭, `C` 아이콘, `F2`, 자동완성 | 같은 상호의 거래처를 혼동하지 않도록 사업자번호 확인 |
| 하단부 거래처 | 상단부 거래처가 하단부로 반영될 수 있음 | 카드 매입처럼 하단 분개 거래처가 다른 경우 수정 필요 |

거래처는 반드시 증빙상의 공급자 또는 공급받는자와 맞아야 한다. 매출전표에서는 공급받는자, 매입전표에서는 공급자를 정확히 입력한다.

## 유형 입력

`유형`은 매입매출전표의 가장 중요한 세무 코드다. 유형에 따라 부가세율, 신고서 반영 위치, 세금계산서/계산서/카드/현금영수증 집계가 결정된다.

### 매출 유형

| 코드 | 명칭 | 거래증빙 | 사용 상황 | 부가세 |
|---:|---|---|---|---|
| 11 | 과세 | 세금계산서 | 과세 전자/종이 세금계산서 발급 매출 | 10% |

## 품명, 수량, 단가

품명, 수량, 단가는 세금계산서의 공급 품목 정보를 구성한다. 입력한 품명은 전자세금계산서 발행과 적요, 복수거래 내역에 영향을 줄 수 있다.

| 항목 | 설명 | 주의점 |
|---|---|---|
| 품명 | 거래 품목 또는 용역명 | 증빙상의 품명과 다르면 발행 문서와 내부 적요가 어긋날 수 있음 |
| 수량 | 거래 수량 | 소수점 자리수는 회사 환경설정과 세금계산서 발행 기준 확인 |
| 단가 | 단위당 금액 | 수량과 단가 곱으로 공급가액 자동 계산 |
| 공급가액 | 부가세 제외 금액 | 직접 입력 가능 |
| 부가세 | 과세유형에 따른 세액 | 과세 거래는 자동 계산, 단수 차이는 직접 수정 가능 |
| 복수거래 | 품목이 2개 이상일 때 사용 | 최대 99개까지 입력 가능, 세금계산서 인쇄에는 일부 품목만 표시될 수 있음 |

영세와 면세 유형은 부가세가 표시되지 않는다. 과세 거래에서 세금계산서 원본의 세액과 프로그램 계산 세액이 다르면 단수 차이를 조정해야 한다.

## 전자 항목

`전자` 항목은 세금계산서 또는 계산서가 전자 형태인지, 그리고 WEHAGO에서 직접 발행할 것인지 판단하는 필드다.

| 입력 상태 | 의미 | 신고자료 영향 |
|---|---|---|
| 공란 | WEHAGO에서 전자발행 기능으로 발행할 수 있는 상태 | 발행 전에는 전자발행 처리 필요 |
| `1.전자입력` | 이미 전자세금계산서를 수취했거나 다른 사이트에서 발급해 발행 절차를 생략하는 상태 | 세금계산서합계표 또는 계산서합계표의 전자분으로 집계 |
| 전자발행 상태 | WEHAGO에서 발행 처리된 상태 | 전송결과와 국세청 신고 상태 확인 필요 |

전자 여부를 잘못 입력하면 전자분과 전자외분 집계가 어긋난다. 특히 과세기간 종료일 다음달 11일 이내 국세청 전송 여부는 합계표의 전자/전자외 구분과도 연결된다.

## 전자세금계산서 발행

전자발행은 선택한 매출 전표를 전자세금계산서 또는 전자계산서로 발행하는 기능이다. 기본적으로 매출 유형 `11.과세`, `12.영세`, `13.면세`가 발행 대상이다.

| 단계 | 처리 내용 | 확인 포인트 |
|---:|---|---|
| 1 | 매입매출전표 상단부에 거래처, 유형, 품명, 공급가액, 부가세를 입력한다. | 거래처 사업자번호, 이메일, 공급가액, 세액 확인 |
| 2 | 전자 항목을 공란으로 둔다. | 이미 다른 경로로 발급한 건이면 `1.전자입력` 사용 |
| 3 | 전자발행 기능을 실행한다. | 발행 대상 유형과 선택 전표 확인 |
| 4 | 발행상태와 처리상태를 확인한다. | 확인요청, 발행요청, 전송성공, 전송실패 등 상태 확인 |
| 5 | 전송결과 또는 전자세금계산서 조회로 국세청 전송 상태를 확인한다. | 신고서 반영 전 누락·실패 건 점검 |

전자세금계산서가 발행되면 삭제와 수정이 제한될 수 있다. 잘못 발행한 경우에는 단순 전표 수정이 아니라 발행취소, 수정세금계산서, 국세청 전송상태 확인이 필요하다.

## 분개 입력

매입매출전표의 `분개` 항목은 상단 세무자료를 어떤 회계분개로 장부에 반영할지 결정한다.

| 코드 | 분개유형 | 의미 | 하단부 영향 |
|---:|---|---|---|
| 0 | 분개없음 | 세무자료만 입력하고 분개를 만들지 않음 | 하단부 분개 없음 |
| 1 | 현금 | 현금 입출금으로 처리 | 현금 계정 중심으로 자동 반영 |
| 2 | 외상 | 외상매출금 또는 외상매입금으로 자동분개 | 해당 계정은 직접 수정 불가 |
| 3 | 혼합 | 현금, 외상, 다른 계정을 혼합해 직접 조정 | 하단부 계정과 금액을 유연하게 입력 |
| 4 | 카드 | 카드 결제 또는 카드채권/카드채무 성격으로 처리 | 카드 관련 계정과 거래처 확인 필요 |
| 5 | 외상추가 | 추가 외상 계정 설정 사용 | 분개계정과목설정 확인 |
| 6 | 외상추가2 | 추가 외상 계정 설정 사용 | 분개계정과목설정 확인 |
| 7 | 외상추가3 | 추가 외상 계정 설정 사용 | 분개계정과목설정 확인 |

`2.외상`을 선택하면 외상매출금 또는 외상매입금으로 자동분개되고 해당 계정을 바꾸기 어렵다. 다른 채권·채무 계정, 선수금, 미수금, 카드채권, 수수료 등을 함께 반영해야 한다면 `3.혼합`을 선택하는 편이 안전하다.

하단부 입력자료는 계정별원장, 분개장, 총계정원장, 재무제표로 이어진다. 상단부의 공급가액과 부가세가 세무자료를 만들고, 하단부의 계정과 금액이 회계장부를 만든다고 보면 된다.

## 매출 세금계산서 발급 후 신용카드 결제

세금계산서를 먼저 발급한 매출을 신용카드로 결제받으면, 같은 매출이 세금계산서 매출과 카드매출로 이중 반영될 위험이 있다. 핵심은 카드 결제를 새로운 매출로 다시 잡지 않고, 이미 발급한 세금계산서 매출의 결제수단으로 처리하는 것이다.

| 상황 | 처리 방향 | 확인 포인트 |
|---|---|---|
| 세금계산서 발급과 카드결제가 같은 날 처리 | 매입매출전표에서 세금계산서 유형과 카드 분개유형을 함께 고려 | 신용카드매출전표발행 관련 집계에서 세금계산서 발급분으로 구분되는지 확인 |
| 세금계산서 발급 후 다른 날 카드로 결제 | 일반전표에서 미수금 또는 외상매출금 상계와 신용카드 관리정보를 입력 | 부가세 매출이 중복 반영되지 않게 처리 |
| 카드매출로 별도 입력해버린 경우 | 중복 매출 후보로 검토 | 세금계산서합계표, 신용카드매출 집계, 부가세신고서 반영 확인 |

실무 체크 포인트는 다음과 같다.

- 세금계산서 매출은 이미 과세표준에 반영된 거래다.
- 카드 결제는 매출 발생이 아니라 채권 회수 또는 결제수단 변경에 가깝다.
- 신용카드매출전표발행 관련 서식에서는 세금계산서 발급분과 카드 매출분을 구분해야 한다.
- 같은 거래처, 같은 금액, 같은 일자에 세금계산서와 카드 입력이 모두 있으면 중복 여부를 반드시 검토한다.

## 관리항목

관리항목은 매입매출전표를 부서, 사원, 현장, 프로젝트, 자금, 주류 여부 등으로 구분해 후속 장부와 신고자료를 더 세밀하게 조회하기 위한 값이다.

| 관리항목 | 입력 목적 | 후속 활용 |
|---|---|---|
| 사원 | 담당자 또는 책임자 기준 관리 | 사원별 매입매출장, 영업 담당자별 실적 |
| 부서 | 부서별 매출·매입·비용 귀속 | 부서별 손익, 부서별 원장 |
| 현장 | 공사·현장별 매입매출 관리 | 현장별 원가, 현장별 매입매출장 |
| 프로젝트 | 프로젝트별 수익·비용 관리 | 프로젝트 손익, 정산 |
| 자금관리 | 자금계획 또는 수금·지급 예정 관리 | 자금계획, 채권채무 관리 |
| 주류 여부 | 주류 관련 거래 구분 | 주류업종 또는 관련 부속 집계 |

관리항목은 입력 화면에서 보이더라도 회사 환경설정과 계정과목별 관리항목 설정이 맞아야 정상적으로 사용할 수 있다. 입력 후에는 매입매출장의 현장별, 프로젝트별, 부서별, 사원별 탭에서 조회되는지 확인한다.

## 입력 전 체크리스트

- 이 거래가 일반전표가 아니라 매입매출전표에 입력해야 하는 부가세 관련 거래인지 확인한다.
- 증빙 종류가 세금계산서, 계산서, 카드, 현금영수증, 수입세금계산서 중 무엇인지 확인한다.
- 거래처코드와 사업자등록번호가 거래처등록에 정확히 등록되어 있는지 확인한다.
- 매출인지 매입인지 먼저 구분한다.
- 과세, 영세, 면세, 불공제, 카드, 현금영수증 등 유형 코드를 증빙에 맞게 고른다.
- 전자세금계산서를 WEHAGO에서 발행할 것인지, 이미 발급·수취한 전자분으로 입력할 것인지 결정한다.
- 분개유형을 현금, 외상, 혼합, 카드 중 무엇으로 처리할지 정한다.
- 현장, 프로젝트, 부서, 사원 등 관리항목이 필요한 거래인지 확인한다.

## 입력 후 검토 체크리스트

- 상단부 공급가액과 부가세가 증빙 원본과 일치하는지 확인한다.
- 영세 또는 면세 거래에 부가세가 잘못 들어가지 않았는지 확인한다.
- 전자 항목이 전자분과 전자외분 집계에 맞게 들어갔는지 확인한다.
- 하단부 분개 차변/대변 합계가 맞는지 확인한다.
- `2.외상` 분개유형 사용 시 외상매출금 또는 외상매입금 계정이 의도한 계정인지 확인한다.
- 카드 결제 거래가 세금계산서 매출과 중복으로 잡히지 않았는지 확인한다.
- 매입매출장, 분개장, 세금계산서합계표 또는 계산서합계표에 의도대로 반영되는지 확인한다.
- 전자발행 건은 전송결과와 국세청 상태를 확인한다.

## BK 설계 반영 메모

이 문서는 위하고 사용법 정리이며 BK의 공식 스펙은 아니다. 다만 현재 `kat_bk` 설계와 연결할 때는 아래 포인트를 참고할 수 있다.

- BK의 매입매출전표는 상단 세무헤더와 하단 회계분개 라인의 2계층 엔티티로 모델링하는 것이 적합하다.
- 과세유형 `11~25`, `51~64`는 `taxType` 코드 마스터로 관리하고, 세율과 신고서 반영 위치는 버전 있는 세법 룰로 관리해야 한다.
- 거래처 사업자등록번호는 세금계산서합계표와 계산서합계표의 집계 키이므로, 거래처 마스터 검증과 휴폐업 검증이 필요하다.
- 전자세금계산서 상태는 전표 상태와 분리해 `전자입력`, `전자발행`, `국세청성공`, `전송실패` 등 외부 연동 상태값으로 보관해야 한다.
- 분개유형은 자동분개 엔진의 입력값이다. 특히 `외상` 자동분개 계정의 수정 제한과 `혼합` 사용 기준을 UI에서 명확히 안내해야 한다.
- 세금계산서 발급 후 카드결제는 중복 매출 방지 규칙이 필요하다. 결제수단별 집계와 세금계산서 발급분 차감 로직을 별도 검증으로 두는 것이 안전하다.
- 관리항목은 사업장, 코스트센터, 프로젝트, 현장, 사원 차원과 매핑하고, 신고자료와 관리회계 조회에서 일관되게 사용해야 한다.
- 전자발행, 신고자료 생성, 자동분개, 원장 반영은 외부 효과와 장부 효과가 섞이므로 멱등키, outbox, 감사로그가 필요하다.
- 마감된 과세기간의 매입매출전표 수정은 일반 update/delete가 아니라 마감해제, 수정세금계산서, 취소전표, 역분개 절차와 연결되어야 한다.
