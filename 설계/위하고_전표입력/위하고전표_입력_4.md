# 위하고 일반전표입력(4) 영상 상세 정리

## 문서 성격

- 대상 영상: [[WEHAGO l Smart A 10] 일반전표입력(4)](https://www.youtube.com/watch?v=Qwpag_ADQec)
- 채널: 위하고(WEHAGO) 공식 채널
- 영상 길이: 10분 37초
- 확인일: 2026-06-24
- 정리 방식: 공개 영상의 원문 대사 전재가 아니라, 영상 설명란의 시간 구간, 화면 흐름, WEHAGO 공식 고객센터 문서, `kat_bk` 내 Smart A 분석 문서를 바탕으로 기능 중심으로 재구성한 상세 노트다.
- 자막 상태: 확인 시점 기준 공개 자막/자동 자막을 확보할 수 없어, 문장 단위 전사가 아니라 화면 기능과 공식 문서 기준의 상세 절차로 정리했다.

## 핵심 요약

4편은 `일반전표입력` 화면의 입력 칸 자체보다, 전표 입력을 검토하고 운영하기 위한 툴바, 더보기 기능, 가이드 영역을 설명한다. 앞선 1~3편이 실제 전표 입력과 관리항목을 다뤘다면, 4편은 입력된 전표를 어떻게 점검하고, 화면을 어떻게 맞추고, 자동전표와 마감 상태를 어떻게 통제하는지에 초점이 있다.

영상은 다음 흐름을 다룬다.

1. 일반전표입력 메뉴의 툴바와 가이드영역 개요
2. 툴바의 `차액전표보기`
3. 툴바의 `행고정`
4. 툴바의 `자동전표검증`
5. 툴바의 `데이터정렬`
6. 더보기 기능의 `중복전표검색`
7. 더보기 기능의 `분개설정`
8. 더보기 기능의 `화면구성`
9. 더보기 기능의 `자동분개 일괄검색 및 삭제`
10. 더보기 기능의 `일마감`
11. 안내 메시지 영역의 거래처코드 입력방법 설정
12. 공통 단축키 안내
13. 일정관리
14. 작성자
15. 메모

핵심은 일반전표입력 화면이 단순 입력 그리드가 아니라는 점이다. 대차 차액, 중복 전표, 자동전표 미확정, 마감 잠금, 작성자, 메모, 일정 같은 운영 정보가 같은 화면 안에 붙어 있고, 이 정보들이 전표 검토와 마감 품질을 좌우한다.

## 툴바 기능 상세

### 차액전표보기

`차액전표보기`는 대차 차액이 발생한 전표를 찾아 보여주는 기능이다. 일반전표에서 대체전표는 같은 전표번호 안에서 차변 합계와 대변 합계가 일치해야 한다. 차액이 남아 있으면 원장과 재무제표에 정상 반영하기 어렵고, 마감 전 검토 대상이 된다.

| 항목 | 내용 |
|---|---|
| 단축키 | `Shift+F10` |
| 찾는 대상 | 차변 합계와 대변 합계가 맞지 않는 전표 |
| 주 사용 시점 | 대체전표 입력 후, 전표라인삽입 후, 전표묶기/분리 후, 마감 전 점검 |
| 확인 포인트 | 차액 원인, 빠진 상대계정, 금액 입력 위치, 잘못된 구분 코드 |

차액전표가 발견되면 다음 순서로 확인한다.

1. 전표번호 단위로 차변 합계와 대변 합계를 확인한다.
2. 구분 코드가 `3.차변`, `4.대변`으로 맞게 들어갔는지 본다.
3. 금액이 차변 칸과 대변 칸 중 올바른 위치에 입력됐는지 본다.
4. 전표라인 삽입 후 빈 라인이 남아 있거나 상대 계정이 빠졌는지 본다.
5. 분개설정의 금액자동분개 또는 차·대변 변환 설정이 의도대로 동작했는지 확인한다.

BK 설계에서는 차액전표보기를 단순 조회 기능이 아니라 `journal-balance` 검증 결과로 다루는 편이 안전하다. 승인 요청, 기표, 마감 처리 전에는 차대 불일치를 차단해야 한다.

### 행고정

`행고정`은 화면 스크롤 중에도 기준 행을 유지하기 위한 기능이다. 검색된 전표의 첫 행부터 현재 커서가 위치한 행까지 고정하고, 해제 기능으로 다시 일반 스크롤 상태로 되돌린다.

| 활용 상황 | 설명 |
|---|---|
| 전표가 많은 기간 검토 | 월말처럼 전표가 많을 때 위쪽 기준 데이터를 유지한 채 아래쪽 전표를 비교 |
| 특정 전표 묶음 비교 | 같은 일자 또는 같은 거래처의 전표를 보면서 아래쪽 라인을 확인 |
| 검토 기준 유지 | 제목 행 또는 기준 전표를 화면에 남겨 둔 채 입력 오류를 찾음 |

행고정은 데이터 자체를 바꾸지 않는 화면 보조 기능이다. 다만 고정된 영역 때문에 실제 선택된 행을 착각할 수 있으므로 삭제, 일괄적용, 전표묶기 같은 작업 전에는 체크된 행을 다시 확인해야 한다.

### 자동전표검증

`자동전표검증`은 자동전표처리 메뉴에서 일반전표입력으로 넘어온 전표의 확정 상태를 확인하는 기능이다. WEHAGO 고객센터 기준으로 현재 회계기간의 신용카드, 현금영수증, 통장 메뉴에서 전송한 전표 중 `전표확정`이 아닌 경우를 검증한다.

| 원천 | 검증 의미 | 확인 포인트 |
|---|---|---|
| 신용카드 자동전표 | 카드 수집 자료가 일반전표로 전송됐지만 확정되지 않은 상태 확인 | 카드 사용내역 중복, 계정 추천 상태, 증빙 분류 |
| 현금영수증 자동전표 | 현금영수증 수집 자료의 전표 반영 상태 확인 | 거래처, 과세/비과세, 비용 계정 |
| 통장 자동전표 | 통장 거래내역 기반 자동분개 전표 상태 확인 | 입출금 성격, 상대 계정, 거래처, 중복 수기 입력 |

자동전표검증은 자동수집 자료를 신뢰하기 전에 거치는 점검 단계다. 자동전표가 확정되지 않았는데 마감 또는 보고를 진행하면, 원장 금액과 원천 수집자료의 상태가 맞지 않을 수 있다.

BK 설계에서는 자동전표검증을 `integrations`와 `journals` 사이의 상태 검증으로 볼 수 있다. 외부 수집자료는 원천 수집 상태, 추천 분개 상태, 전표 전송 상태, 승인/기표 상태를 분리해 추적해야 한다.

### 데이터정렬

`데이터정렬`은 현재 조회된 전표의 표시 순서를 바꾼다. 정렬 기준은 입력순, 날짜순, 거래처코드순, 계정코드순, 계정과목명순 등이 있다.

| 정렬 기준 | 사용 목적 | 주의점 |
|---|---|---|
| 입력순 | 사용자가 입력한 순서대로 확인 | 입력 순서와 회계일자가 다를 수 있음 |
| 날짜순 | 거래일자 흐름대로 검토 | 같은 일자 안의 전표번호와 묶임 상태 확인 필요 |
| 거래처코드순 | 특정 거래처별 입력 누락 또는 중복 확인 | 거래처코드 미입력 전표는 별도 확인 필요 |
| 계정코드순 | 계정별 입력 오류, 비용 계정 검토 | 같은 계정이라도 거래처와 관리항목이 다를 수 있음 |
| 계정과목명순 | 계정명 기준으로 사람이 읽기 쉽게 확인 | 코드 체계와 다른 순서로 보일 수 있음 |

정렬은 조회 편의 기능이지만 일괄 처리와 함께 쓰면 위험할 수 있다. 정렬 후에는 선택 체크 상태가 의도한 전표에 남아 있는지 확인해야 한다.

## 더보기 기능 상세

### 중복전표검색

`중복전표검색`은 같은 거래가 두 번 들어갔을 가능성을 찾는 기능이다. 조건은 일자, 유형, 계정, 거래처, 금액 중에서 선택한다. 단축키는 `Shift+F7`이다.

| 조건 | 중복 의심 예 | 검토 포인트 |
|---|---|---|
| 일자 | 같은 날짜에 같은 거래가 반복 입력 | 실제 반복 거래인지, 중복 입력인지 확인 |
| 유형 | 같은 전표 구분으로 반복 입력 | 현금전표와 대체전표가 섞였는지 확인 |
| 계정 | 같은 계정으로 같은 금액 반복 | 비용 중복, 수수료 중복, 자동분개 중복 |
| 거래처 | 같은 거래처와 같은 금액 반복 | 거래처별 원장과 증빙 대조 |
| 금액 | 동일 금액이 여러 번 등장 | 금액만 같고 거래가 다른 경우도 있으므로 적요와 증빙 확인 |

중복전표검색 결과가 나왔다고 해서 무조건 삭제하면 안 된다. 매월 반복 비용, 같은 금액의 여러 영수증, 같은 거래처의 다건 거래는 정상일 수 있다. 삭제 전에는 증빙번호, 거래일자, 적요, 자동전표 원천을 함께 확인해야 한다.

BK에서는 중복전표검색을 단순 조건검색보다 한 단계 높은 검증 기능으로 설계할 수 있다. `sameDate + samePartner + sameAmount + sameEvidenceKey` 같은 후보 규칙을 두고, 사용자가 정상 반복거래로 확인한 경우 예외 메모를 남길 수 있어야 한다.

### 분개설정

`분개설정`은 일반전표 입력 중 반복 입력을 줄이기 위한 보조 설정이다. 공식 문서 기준으로 금액자동분개, 차·대변 변환, 거래처코드 복사 설정을 다룬다.

| 설정 | 의미 | 사용 예 | 주의점 |
|---|---|---|---|
| 금액자동분개 | 상대계정 입력 시 차액을 자동으로 분개 | 차변 금액을 먼저 입력하고 대변 상대계정 입력 시 남은 금액 반영 | 잘못된 금액이 자동 반영될 수 있으므로 소계 확인 필요 |
| 차·대변 변환 | 대체전표 입력 중 차액 발생 시 차변/대변 방향을 보조 변환 | 상대 라인의 구분을 자동으로 맞추는 흐름 | 의도와 다른 방향으로 잡히지 않았는지 확인 |
| 거래처코드 | 거래처 Code 칸에서 엔터 시 거래처코드 또는 거래처명 복사 | 상단 거래처 정보를 하단 라인에 빠르게 반영 | 거래처가 필요 없는 계정이나 다른 거래처 라인에 잘못 복사될 수 있음 |

분개설정은 입력 속도를 높이는 기능이지만 회계 판단을 대신하지 않는다. 자동으로 들어간 상대금액과 거래처가 실제 증빙과 맞는지 마지막에 확인해야 한다.

### 화면구성

`화면구성`은 일반전표입력 화면의 표시와 기본 입력 방식을 조정한다. 단축키는 `Ctrl+A`이다.

| 설정 | 의미 | 후속 영향 |
|---|---|---|
| 전표번호보기 | 일반전표입력 화면의 번호 칸 표시 여부 | 전표묶기, 전표분리, 인쇄 전 확인 편의 |
| 기본화면구성 | 입력 화면의 기본 배치와 기준 | 거래처원장, 계정별원장 등 원장 메뉴의 전표수정 화면구성에도 반영될 수 있음 |
| 자동분개 설정 | 기본화면구성 기준의 자동완성 동작 | 과거 입력 패턴 또는 화면 기준에 따른 추천·자동완성에 영향 |

전표번호를 숨기면 화면은 단순해지지만, 여러 라인이 같은 전표인지 확인하기 어려울 수 있다. 전표 검토, 묶기/분리, 인쇄가 잦은 사용자라면 전표번호 표시가 유리하다.

BK 설계 관점에서는 화면구성을 사용자별 UI preference와 회사별 회계정책 설정으로 분리해야 한다. 예를 들어 전표번호 표시 여부는 사용자 화면 설정이지만, 자동분개 기준과 관리항목 필수 여부는 회사 정책 또는 테넌트 설정에 가깝다.

### 자동분개 일괄검색 및 삭제

`자동분개 일괄검색 및 삭제`는 회계, 급여, 물류 모듈의 각 전표처리 메뉴에서 전송한 전표 데이터를 기간과 삭제 항목 기준으로 검색하고 삭제하는 기능이다. 단축키는 `Shift+F6`이다.

| 항목 | 내용 |
|---|---|
| 검색 대상 | 회계, 급여, 물류 등 다른 메뉴에서 전표전송된 자동분개 데이터 |
| 검색 기준 | 기간, 삭제 또는 조회할 전표처리 항목 |
| 주요 사용 상황 | 잘못 전송된 자동전표 정리, 원천 모듈 재전송 준비, 기간별 자동전표 확인 |
| 위험 지점 | 원천 모듈 상태와 일반전표 삭제 상태가 어긋날 수 있음 |

삭제 전에는 다음을 확인해야 한다.

1. 자동전표 원천 메뉴가 무엇인지 확인한다.
2. 기간이 정확한지 확인한다.
3. 삭제 대상이 수기 입력 전표가 아니라 자동전송 전표인지 확인한다.
4. 삭제 후 원천 메뉴에서 재전송이 필요한지 확인한다.
5. 마감된 기간 또는 승인된 전표가 포함되어 있지 않은지 확인한다.

BK에서는 자동분개 일괄삭제를 실제 삭제가 아니라 원천 전송 상태와 전표 상태를 함께 되돌리는 보정 이벤트로 설계하는 편이 좋다. 이미 원장에 반영된 전표는 물리 삭제보다 취소, 반려, 역분개, 재전송 이력으로 처리해야 추적성이 남는다.

### 일마감

`일마감`은 특정 일자를 기준으로 그 이전 자료의 입력, 수정, 삭제를 제한하는 기능이다. 공식 문서 기준으로 마감해제를 통해 해제할 수 있다.

| 상태 | 의미 | 허용/제한 |
|---|---|---|
| 마감 전 | 일반 입력과 수정 가능 | 회사 권한과 전표 상태에 따라 처리 |
| 일마감 | 마감일자 이전 자료 잠금 | 입력, 수정, 삭제 제한 |
| 마감해제 | 잠금을 해제 | 권한, 사유, 감사로그 필요 |

일마감은 월마감이나 결산마감보다 작은 단위의 통제지만, 회계 품질에는 중요하다. 마감 이후 수정을 허용하면 이미 검토한 자료가 바뀔 수 있으므로, 마감해제는 승인과 사유 기록이 필요하다.

BK 설계에서는 일마감을 `closing` 도메인의 잠금 상태로 모델링해야 한다. 전표 저장, 수정, 삭제, 자동전표 전송, 코드변환, 전표이동은 모두 마감 잠금을 확인해야 한다.

## 가이드영역 상세

### 안내 메시지 영역

안내 메시지 영역은 마우스 커서가 위치한 항목에 대한 기본 가이드를 제공한다. 입력자가 어떤 칸에 커서를 두었는지에 따라 계정과목, 거래처, 적요, 관리항목, 단축키 안내가 달라진다.

| 커서 위치 | 안내 내용 예 | 입력자에게 주는 도움 |
|---|---|---|
| 거래일자 | 거래전표의 일자를 입력하라는 안내 | 증빙일자와 회계일자 확인 |
| 거래처 | 거래처 코드도움, 자동완성, 코드 표시 방식 안내 | 거래처 미등록 또는 코드 누락 방지 |
| 계정과목 | 계정과목 코드도움, 계정명 검색 안내 | 계정 선택 오류 감소 |
| 적요 | 적요코드 도움, 적요항목 편집 안내 | 반복 적요 표준화 |
| 관리항목 | 부서, 사원, 현장, 프로젝트, 자금 등 입력 안내 | 보조부 누락 방지 |

안내 영역은 초보자용 설명에 그치지 않는다. 전표 입력 중 어떤 필드가 원장, 보조장부, 신고서식에 영향을 주는지 즉시 알려 주는 역할을 한다.

### 거래처코드 입력방법 설정

거래처코드 입력방법은 일반전표의 거래처 관리 품질에 직접 영향을 준다. 공식 문서 기준으로 거래처코드 보이기 설정에 따라 거래처 Code 칸 표시와 자동완성 사용 여부가 달라진다.

| 설정/상황 | 의미 | 실무 영향 |
|---|---|---|
| 거래처코드 보이기 ON | 거래처 Code 칸을 표시하고 자동완성 설정 가능 | 거래처원장, 채권채무 관리에 유리 |
| 거래처코드 보이기 OFF | 거래처 코드 칸이 제거되고 자동완성 비활성 또는 제한 | 화면은 단순하지만 코드 기반 추적이 약해짐 |
| 거래처명만 입력 | Code 없이 상호만 직접 입력 | 거래처원장에서 거래처별 조회가 어렵고 미등록거래처로 조회될 수 있음 |
| 분개설정 거래처코드 복사 | 엔터 입력 시 상단 거래처 정보를 하단 라인으로 복사 | 반복 입력은 빠르지만 잘못 복사된 거래처 확인 필요 |

거래처를 코드 없이 이름만 입력하면 장부에는 보이더라도 거래처별 원장, 채권채무 잔액, 미수/미지급 관리에서는 추적성이 떨어진다. BK에서는 거래처가 필요한 계정에 대해 거래처코드를 필수로 검증하는 규칙이 필요하다.

### 공통 단축키 안내

공통 단축키는 환경설정의 단축키 설정을 사용으로 둔 뒤 활용한다. 메뉴별 단축키는 툴바 버튼에 마우스를 올리거나 더보기 기능모음에서 확인할 수 있다.

| 단축키 | 기능 | 비고 |
|---|---|---|
| `F2` | 코드도움 | 거래처, 계정과목, 적요 등 조회 |
| `F4` | 전표복사 | 선택 전표를 다른 일자로 복사 |
| `F5` | 삭제 | 선택 전표 삭제 |
| `F7` | 전표묶기 | 전표 묶기, 분리, 순서 조정 |
| `F9` | 인쇄 | 선택 전표 출력 |
| `Shift+F7` | 중복전표검색 | 일자, 유형, 계정, 거래처, 금액 기준 |
| `Shift+F10` | 차액전표보기 | 대차 불일치 전표 조회 |
| `Shift+F6` | 자동분개 일괄검색 | 자동전표 검색 및 삭제 |
| `Ctrl+A` | 화면구성 | 전표번호보기, 기본화면구성 등 |
| `Ctrl+M` | 전표이동 | 선택 전표를 지정 일자로 이동 |
| `Insert` | 전표라인삽입 | 같은 전표번호의 빈 라인 삽입 |

단축키는 반복 입력에는 강력하지만, 삭제와 이동처럼 데이터에 영향을 주는 기능도 포함한다. 권한과 상태에 따라 단축키 실행 가능 여부를 제한하는 것이 안전하다.

### 일정관리

가이드영역의 일정 영역에서는 WEHAGO 일정관리 서비스에 입력된 정보를 확인할 수 있다. 달력의 일정을 클릭하고 더보기로 일정관리 서비스를 실행할 수 있다.

| 활용 상황 | 예시 |
|---|---|
| 월마감 일정 확인 | 특정일까지 전표 입력 마감, 검토 마감, 세무 신고 자료 마감 |
| 증빙 회수 일정 | 거래처별 증빙 요청 또는 미수 증빙 확인 일정 |
| 내부 검토 일정 | 담당자 검토, 승인자 확인, 마감해제 승인 일정 |
| 신고 일정 | 부가세, 원천세, 법인세 등 신고 관련 준비 일정 |

전표 입력 화면에서 일정을 함께 볼 수 있으면 입력 작업과 마감 작업을 분리하지 않고 관리할 수 있다. BK에서는 일정 자체를 만들지 않더라도 마감 캘린더, 업무 할당, 검토 기한과 연결할 수 있다.

### 작성자

작성자 영역은 커서가 위치한 전표의 작성자 정보와 최종 입력 시간을 보여준다. 이는 누가 언제 전표를 입력하거나 수정했는지 확인하는 최소한의 운영 정보다.

| 표시 정보 | 의미 | 활용 |
|---|---|---|
| 작성자 | 전표를 입력한 사용자 | 오류 문의, 담당자 확인, SOD 검토 |
| 최종입력시간 | 마지막 입력 또는 수정 시각 | 마감 직전 변경 추적, 검토 순서 판단 |
| 연동정보와 함께 확인 | 전표복사/이동 또는 타 메뉴 전표처리 정보 | 수기 입력인지 자동전송인지 구분 |

BK에서는 작성자 표시를 넘어 `createdBy`, `updatedBy`, `postedBy`, `approvedBy`, `sourceSystem`, `sourceReference`를 분리해야 한다. 작성자와 승인자가 같은 경우를 차단하는 SOD 규칙도 필요하다.

### 메모

메모 영역은 전표에 대한 상세 내용을 남기는 곳이다. 공식 문서 기준으로 입력한 메모는 조건검색에서 메모검색으로 다시 찾을 수 있다.

| 메모 사용 예 | 내용 |
|---|---|
| 증빙 확인 | 원본 영수증 미도착, 거래처 확인 필요, 세금계산서 재발행 요청 |
| 회계 판단 | 자산화 여부 검토, 비용 귀속 부서 확인, 계정 변경 후보 |
| 마감 이슈 | 마감 전 확인 필요, 다음 월 조정 예정, 승인 보류 사유 |
| 자동전표 이슈 | 통장 자동분개 중복 의심, 카드 자료 재수집 필요 |
| 고객 확인 요청 | 이용회사 담당자에게 확인 필요한 거래 |

메모는 장부의 공식 적요와 다르다. 적요는 거래 내용을 장부에 남기는 필드이고, 메모는 검토와 운영을 위한 내부 설명에 가깝다. 따라서 신고서나 원장에 노출될 문구와 내부 검토 문구를 구분해야 한다.

## 기능별 검토 체크리스트

### 입력 후 검토

- `차액전표보기`로 대차 불일치 전표가 없는지 확인한다.
- `중복전표검색`으로 같은 증빙이 이중 입력되지 않았는지 확인한다.
- `자동전표검증`으로 자동전표처리에서 넘어온 자료 중 미확정 상태가 없는지 확인한다.
- `선택전표소계`에서 커서 위치 전표번호의 차변/대변 합계를 확인한다.
- 거래처코드 없이 이름만 입력된 전표가 없는지 확인한다.

### 일괄 처리 전

- 현재 데이터정렬 기준을 확인한다.
- 조건검색이 걸려 있어 일부 전표만 보이는 상태인지 확인한다.
- 체크된 전표가 실제 처리 대상인지 확인한다.
- 자동전표, 증빙, 자금항목, 업무용승용차, 카드계좌 연결 정보가 있는지 확인한다.
- 마감된 기간의 전표가 포함되어 있지 않은지 확인한다.

### 마감 전

- 일마감 대상 일자와 기간을 확인한다.
- 자동전표 미확정 자료를 정리한다.
- 차액전표와 중복전표 후보를 확인한다.
- 작성자와 최종 입력 시간이 비정상적으로 마감 직전에 몰려 있지 않은지 본다.
- 메모에 남은 확인 요청 사항이 해결됐는지 확인한다.

## BK 설계 반영 메모

이 문서는 위하고 사용법 정리이며 BK의 공식 스펙은 아니다. 다만 현재 `kat_bk` 설계와 연결할 때는 아래 포인트를 참고할 수 있다.

- `차액전표보기`는 BK의 `journal-balance` 검증 결과 화면으로 모델링할 수 있다. 승인 또는 기표 전 차대 불일치는 차단해야 한다.
- `자동전표검증`은 외부 수집자료, 자동분개 추천, 전표전송, 전표확정 상태를 분리한 상태머신으로 구현해야 한다.
- `중복전표검색`은 일자, 거래처, 계정, 금액, 증빙키, 원천 시스템 참조값을 조합한 중복 후보 탐지로 확장할 수 있다.
- `분개설정`은 사용자 편의 설정과 회사 회계정책을 구분해야 한다. 금액자동분개는 UI 보조지만 관리항목 필수 여부는 정책이다.
- `화면구성`은 사용자별 preference로 관리하되, 원장 전표수정 화면과 입력 화면이 같은 구성 규칙을 공유할 수 있다.
- `자동분개 일괄검색 및 삭제`는 실제 삭제보다 원천 전송 상태와 전표 상태를 함께 되돌리는 감사 가능한 보정 이벤트가 적합하다.
- `일마감`은 `closing` 도메인의 기간 잠금으로 모델링하고, 입력/수정/삭제/전표이동/코드변환/자동전표 전송 시 잠금 검증을 공통 적용해야 한다.
- `작성자`와 `최종입력시간`은 최소 감사정보이며, BK에서는 작성/수정/승인/기표/마감 행위자를 모두 분리해야 한다.
- `메모`는 내부 검토 메모와 장부 적요를 분리해 저장해야 한다. 검색 가능해야 하지만 보고서 노출 범위는 별도로 통제해야 한다.
- `일정관리`는 마감 캘린더, 업무 할당, 고객 확인 요청 기한과 연결할 수 있다.
- 단축키 실행도 권한과 상태 검증을 우회하면 안 된다. UI 단축키는 결국 동일한 API 권한·마감·감사 검증을 거쳐야 한다.
