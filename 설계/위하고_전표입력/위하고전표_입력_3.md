# 위하고 일반전표입력(3) 영상 상세 정리

## 문서 성격

- 대상 영상: [[WEHAGO l Smart A 10] 일반전표입력(3)](https://www.youtube.com/watch?v=XVz9Lquodb0)
- 채널: 위하고(WEHAGO) 공식 채널
- 영상 길이: 12분 12초
- 확인일: 2026-06-24
- 정리 방식: 공개 영상의 원문 대사 전재가 아니라, 영상 설명란의 시간 구간, 화면 흐름, WEHAGO 공식 고객센터 문서, `kat_bk` 내 Smart A 분석 문서를 바탕으로 기능 중심으로 재구성한 상세 노트다.
- 자막 상태: 확인 시점 기준 공개 자막/자동 자막을 확보할 수 없어, 문장 단위 전사가 아니라 화면 기능과 공식 문서 기준의 상세 절차로 정리했다.

## 핵심 요약

3편은 `일반전표입력`에서 자금관리 성격의 보조항목을 입력하는 방법과, 입력 후 전표를 다룰 때 사용하는 액션바 기능을 설명한다. 1편이 일반전표의 기본 입력, 2편이 관리항목과 세무 부속서류 연결을 다뤘다면, 3편은 받을어음, 지급어음, 차입금처럼 전표 라인만으로 끝나지 않고 별도 자금관리 대장과 연결되는 항목에 초점이 있다.

영상은 다음 흐름을 다룬다.

1. 받을어음 입력 방법
2. 지급어음 입력 방법
3. 차입금 입력 방법
4. 관리항목이 등록된 전표 삭제 방법
5. 액션바 기능 개요
6. 코드변환
7. 일괄적용
8. 전표라인 삽입
9. 전표묶기
10. 전표복사와 전표이동
11. 체크펜

핵심은 자금관리 항목이 붙은 전표는 단순 분개가 아니라는 점이다. 받을어음은 어음번호, 만기, 지급은행, 거래처별 현황으로 이어지고, 지급어음은 어음집계표와 지급어음현황으로 이어지며, 차입금은 차입금 스케줄과 상환 현황으로 이어진다. 따라서 일반 전표보다 삭제, 복사, 이동, 코드변환 시 제한과 검토가 더 필요하다.

## 자금관리 항목의 큰 그림

받을어음, 지급어음, 차입금은 일반전표에 입력되지만 성격상 전표 라인 하나로 끝나지 않는다. 회계 분개와 별도로 만기, 상환, 지급, 회수, 상태 변경을 계속 추적해야 한다.

| 항목 | 전표 입력 의미 | 후속 관리 화면 | 핵심 관리 포인트 |
|---|---|---|---|
| 받을어음 | 거래처로부터 어음을 받아 채권을 어음으로 보유 | 받을어음현황 | 어음번호, 지급은행, 만기일, 거래처, 보유/할인/배서/만기 등 상태 |
| 지급어음 | 회사가 거래처에 어음을 발행하거나 교부 | 지급어음현황, 어음집계표 | 사전 어음등록, 어음번호, 수령/발행 정보, 지급기일, 지급 상태 |
| 차입금 | 금융기관 또는 거래처로부터 자금을 차입 | 차입금스케줄관리, 차입금현황 | 차입처, 차입기간, 원금, 상환방법, 이자율, 상환스케줄 |

전표 입력자는 회계 계정과 금액만 맞추는 것이 아니라 후속 관리대장에 필요한 식별정보를 함께 남겨야 한다. 이 정보가 누락되면 원장 금액은 맞아도 만기별 자금계획, 어음현황, 차입금 상환계획이 맞지 않는다.

## 받을어음 입력 방법

### 업무 상황

받을어음은 거래처로부터 현금 대신 약속어음 또는 전자어음 등을 수취한 경우에 사용한다. 예를 들어 외상매출금을 어음으로 회수하거나, 매출채권이 어음채권으로 대체되는 상황이 해당된다.

### 입력 흐름

| 단계 | 처리 내용 | 확인 포인트 |
|---:|---|---|
| 1 | 일반전표입력에서 거래일자를 입력한다. | 실제 어음 수취일 또는 회계처리 기준일이 맞는지 확인한다. |
| 2 | 받을어음 계정 또는 받을어음 관리항목이 설정된 계정을 입력한다. | 계정과목 설정에 받을어음 관리가 연결되어 있어야 한다. |
| 3 | 거래처를 입력한다. | 어음 발행인 또는 채권 거래처와 원장 거래처가 맞는지 확인한다. |
| 4 | 어음 관리정보를 입력한다. | 어음번호, 지급은행, 발행일, 만기일, 금액 등 식별정보를 누락하지 않는다. |
| 5 | 상대 계정 라인을 입력해 대차를 맞춘다. | 외상매출금 대체, 매출채권 회수, 기타 채권 대체 등 거래 성격에 맞춘다. |
| 6 | 저장 후 받을어음현황에서 조회한다. | 만기일별, 지급은행별, 거래처별 조회에 의도대로 반영됐는지 확인한다. |

### 받을어음 입력 필드

| 필드 | 의미 | 실무 주의점 |
|---|---|---|
| 거래처 | 어음 관련 채권 상대방 | 거래처 코드가 잘못되면 거래처별 채권 잔액과 어음현황이 어긋난다. |
| 어음번호 | 어음을 식별하는 번호 | 중복 번호 입력을 피하고 전자어음인 경우 원천 번호와 일치시킨다. |
| 지급은행 | 만기 지급이 이루어질 금융기관 | 은행별 만기자금 조회와 회수 관리에 영향을 준다. |
| 발행일 | 어음이 발행된 날짜 | 수취일과 다를 수 있으므로 증빙의 발행일을 확인한다. |
| 만기일 | 어음 결제 예정일 | 받을어음현황의 만기일별 조회와 자금계획의 핵심 기준이다. |
| 금액 | 어음 원금 | 전표 금액과 어음 관리금액이 일치해야 한다. |
| 상태 | 보유, 할인, 배서, 만기, 부도 등 진행 상태 | 최초 입력은 보유 상태로 시작하고 이후 이벤트에 따라 바뀐다. |

### 후속 관리

받을어음은 수취 후 다음 이벤트가 발생할 수 있다.

| 이벤트 | 의미 | 전표/관리 영향 |
|---|---|---|
| 보관 | 어음을 회사가 계속 보유 | 만기 도래 전까지 보유금액으로 남는다. |
| 할인 | 금융기관에 할인해 현금화 | 할인료, 수수료, 현금 입금 전표와 연결된다. |
| 배서 | 다른 거래처에 지급수단으로 넘김 | 채무 상계 또는 지급 처리와 연결된다. |
| 만기 결제 | 만기일에 정상 회수 | 받을어음 감소와 예금 증가 전표가 필요하다. |
| 부도 | 만기 결제가 실패 | 부도어음 관리, 대손 검토, 채권 재분류가 필요하다. |

BK 설계 관점에서는 받을어음을 전표 라인의 부속정보가 아니라 별도 `note` 성격의 보조 원장으로 보아야 한다. 전표 라인은 최초 취득, 할인, 배서, 회수, 부도 같은 상태 전환 이벤트와 연결되고, 각 이벤트는 감사로그와 원장 반영 규칙을 가져야 한다.

## 지급어음 입력 방법

### 업무 상황

지급어음은 회사가 매입대금, 외상매입금, 기타 채무를 어음으로 지급할 때 사용한다. 지급어음은 회사가 앞으로 만기일에 지급해야 할 의무이므로, 전표 입력뿐 아니라 어음번호와 만기별 지급예정 관리가 중요하다.

### 사전 준비

지급어음은 일반전표에서 바로 임의 번호를 입력하는 방식으로 관리하면 안 된다. WEHAGO 공식 도움말 기준으로 지급어음 전표를 입력하려면 먼저 `어음집계표` 또는 어음등록 기능에서 지급어음 정보를 등록해야 한다.

| 준비 항목 | 내용 |
|---|---|
| 어음종류 | 어음, 당좌, 가계, 전자어음 등 어음 구분 |
| 금융기관 | 지급어음과 연결되는 은행 또는 금융기관 |
| 시작 어음번호 | 연속 발행하는 경우 시작 번호 |
| 매수 | 등록할 어음 수량 |
| 전자어음 정보 | 전자어음 일괄등록 또는 편집이 필요한 경우 관련 정보 |
| 사용 가능 상태 | 이미 사용, 폐기, 담보 등으로 처리된 번호를 다시 쓰지 않도록 확인 |

### 입력 흐름

| 단계 | 처리 내용 | 확인 포인트 |
|---:|---|---|
| 1 | 어음집계표에서 지급어음을 먼저 등록한다. | 사용할 어음번호와 금융기관, 어음종류가 맞는지 확인한다. |
| 2 | 일반전표입력에서 거래일자를 입력한다. | 어음 교부일 또는 회계처리 기준일을 확인한다. |
| 3 | 지급어음 계정 또는 지급어음 관리항목이 설정된 계정을 입력한다. | 계정과목의 관리항목 설정이 지급어음과 연결되어 있어야 한다. |
| 4 | 등록된 어음정보를 선택해 전표 라인과 연결한다. | 미등록 어음 또는 이미 사용된 어음을 선택하지 않는다. |
| 5 | 상대 계정 라인을 입력한다. | 외상매입금 대체, 비용 지급, 채무 상계 등 거래 성격을 반영한다. |
| 6 | 지급어음현황과 어음집계표에서 반영 여부를 확인한다. | 거래처별, 지급은행별, 만기일별 조회 결과를 확인한다. |

### 지급어음 입력 필드

| 필드 | 의미 | 실무 주의점 |
|---|---|---|
| 어음번호 | 지급어음의 고유 번호 | 어음집계표에 등록된 번호와 연결해야 한다. |
| 거래처 | 어음을 교부받는 상대방 | 채무 거래처와 어음 수취인이 다르면 사유를 남겨야 한다. |
| 지급은행 | 만기 지급은행 | 은행별 지급예정 관리와 자금계획에 영향이 있다. |
| 발행일/교부일 | 어음을 발행하거나 교부한 날짜 | 거래일자와 다를 수 있으므로 증빙 기준을 확인한다. |
| 만기일 | 실제 지급 예정일 | 만기자금 계획과 지급어음현황의 핵심 기준이다. |
| 금액 | 어음 원금 | 전표 금액, 채무 상계금액, 어음 금액이 맞아야 한다. |
| 상태 | 발행, 보관, 지급, 폐기 등 진행 상태 | 지급 완료 전까지 미결 지급의무로 추적한다. |

### 지급어음 후속 관리

지급어음은 발행 후 만기일까지 지급의무로 남는다. 따라서 단순히 전표가 대차균형을 맞췄는지만 보면 안 되고, 어음대장과 만기자금 계획이 함께 맞아야 한다.

| 이벤트 | 의미 | 확인 포인트 |
|---|---|---|
| 발행/교부 | 거래처에 지급어음을 넘김 | 등록된 어음번호와 전표 연결 확인 |
| 지급 | 만기일에 예금 등으로 결제 | 지급어음 감소와 예금 감소 전표 확인 |
| 폐기/취소 | 잘못 등록되었거나 사용하지 않음 | 이미 전표와 연결된 어음인지 확인 |
| 담보/기타 상태 | 지급어음이 특수 상태로 관리됨 | 어음집계표의 상태와 사유 관리 필요 |

BK 설계 관점에서는 지급어음 번호는 재사용과 중복을 막아야 하며, 전표 입력 시점에 어음번호의 사용 가능 상태를 검증해야 한다. 지급 완료, 폐기, 취소 같은 상태 변경은 전표 수정만으로 처리하지 않고 상태 이벤트와 감사로그로 남기는 편이 안전하다.

## 차입금 입력 방법

### 업무 상황

차입금은 금융기관, 대표자, 관계회사, 기타 거래처로부터 자금을 빌린 거래다. 차입금 전표는 원금이 얼마인지뿐 아니라 언제부터 언제까지 차입했는지, 어떤 방식으로 상환하는지, 이자는 어떻게 계산하고 지급하는지까지 관리해야 한다.

### 입력 흐름

| 단계 | 처리 내용 | 확인 포인트 |
|---:|---|---|
| 1 | 일반전표입력에서 차입일 또는 회계처리 기준일을 입력한다. | 입금일, 약정일, 회계처리일이 다르면 회사 기준을 확인한다. |
| 2 | 차입금 계정 또는 차입금 관리항목이 설정된 계정을 입력한다. | 단기차입금, 장기차입금, 관계회사차입금 등 계정 분류가 맞는지 확인한다. |
| 3 | 차입처를 거래처로 입력한다. | 금융기관, 대표자, 특수관계자 등 거래처 구분이 중요하다. |
| 4 | 차입금 관리정보를 입력한다. | 차입금번호, 차입기간, 원금, 상환구분, 이자율, 자금용도 등을 입력한다. |
| 5 | 상대 계정 라인을 입력한다. | 보통 예금 입금, 기존 채무 대체, 자금 유입 계정과 연결된다. |
| 6 | 차입금스케줄관리 또는 차입금현황에서 조회한다. | 상환 예정, 잔액, 이자 지급 일정이 의도대로 잡혔는지 확인한다. |

### 차입금 입력 필드

| 필드 | 의미 | 실무 주의점 |
|---|---|---|
| 차입금번호 | 차입 건별 식별번호 | 같은 금융기관에서 여러 건을 차입할 수 있으므로 건별 구분이 필요하다. |
| 차입처 | 돈을 빌려준 상대방 | 금융기관, 임원, 관계회사 등 분류에 따라 주석과 세무 검토가 달라질 수 있다. |
| 차입기간 | 차입 시작일과 만기일 | 단기/장기 분류, 유동성 대체, 상환스케줄의 기준이다. |
| 원금 | 차입한 원금 | 전표 금액과 관리 원금이 일치해야 한다. |
| 상환구분 | 만기일시상환, 원금균등, 원리금균등 등 | 스케줄 생성과 잔액 계산 방식에 영향이 있다. |
| 이자율 | 약정 이자율 | 이자비용 계산과 미지급이자 검토에 필요하다. |
| 자금용도 | 운전자금, 시설자금 등 | 내부 보고와 금융기관 약정 관리에 쓰인다. |
| 담보/보증 정보 | 담보, 보증, 약정 관련 정보 | 필요한 경우 별도 문서와 연결해 관리한다. |

### 차입금 후속 관리

차입금은 최초 차입 이후 아래 흐름으로 이어진다.

| 이벤트 | 전표 영향 | 관리 영향 |
|---|---|---|
| 최초 차입 | 예금 증가, 차입금 증가 | 차입금번호와 스케줄 생성 |
| 이자 지급 | 이자비용 또는 미지급이자 정리 | 지급일, 이자율, 원천징수 여부 확인 |
| 원금 상환 | 차입금 감소, 예금 감소 | 상환스케줄과 잔액 업데이트 |
| 만기 연장 | 기존 만기 변경 또는 신규 차입 대체 | 약정 변경 이력과 승인 필요 |
| 유동성 대체 | 장기차입금 일부를 유동성장기부채로 대체 | 결산 시점 분류 검토 필요 |

BK 설계 관점에서는 차입금은 단순 계정 잔액이 아니라 상환스케줄을 가진 금융부채다. 전표 라인과 차입금 스케줄이 서로 검증되어야 하고, 이자 지급과 원금 상환은 멱등키를 가진 별도 이벤트로 처리하는 편이 안전하다.

## 관리항목이 등록된 전표 삭제 방법

관리항목이 등록된 전표는 삭제 시 더 조심해야 한다. 전표 라인을 삭제하면 원장 반영만 사라지는 것이 아니라 해당 라인과 연결된 보조 관리 데이터도 끊길 수 있기 때문이다.

| 확인 항목 | 내용 | 삭제 전 체크 |
|---|---|---|
| 자금항목 | 받을어음, 지급어음, 차입금, 미불관리 등 | 후속 현황 메뉴에 반영된 자료인지 확인 |
| 증빙/카드계좌 | 카드, 계좌, 적격증빙 분류 | 증빙명세서나 신고서 반영 여부 확인 |
| 자동전표 | 수집, 자동분개, 외부연계로 생성된 전표 | 원천 데이터와 재처리 방식 확인 |
| 일마감 | 특정 일자 이전 입력/수정/삭제 제한 | 마감 해제 권한과 사유 필요 |
| 승인/기표 상태 | 검토 또는 승인된 전표 | 단순 삭제보다 취소, 반려, 역분개 흐름 검토 |

실무적으로는 다음 순서가 안전하다.

1. 삭제할 전표의 거래일자, 전표번호, 거래처, 계정, 금액을 확인한다.
2. 관리항목 영역에 받을어음, 지급어음, 차입금, 카드계좌, 업무용승용차 등 연결 정보가 있는지 확인한다.
3. 해당 전표가 후속 현황 메뉴에 반영되어 있는지 확인한다.
4. 일마감, 월마감, 승인 상태가 걸려 있으면 삭제 대신 상태 변경 절차를 따른다.
5. 삭제 후 원장, 보조원장, 자금관리 현황의 잔액이 맞는지 다시 확인한다.

BK에서는 이 흐름을 단순 `delete`로 처리하면 안 된다. 임시저장 전표는 삭제할 수 있더라도, 승인 또는 기표된 전표는 취소전표, 역분개, 마감해제 승인 같은 절차로 분리해야 한다. 모든 변경은 감사로그에 남겨야 한다.

## 액션바 기능 정리

액션바는 선택된 전표 또는 라인을 대상으로 반복 작업을 빠르게 수행하는 도구 영역이다. 편리하지만 선택 범위가 잘못되면 여러 전표가 한 번에 바뀌므로, 실행 전 체크 상태와 조회 조건을 먼저 확인해야 한다.

| 기능 | 단축키/위치 | 기능 요약 | 주의점 |
|---|---|---|---|
| 코드변환 | `Shift+F3` | 거래처, 계정과목, 사원, 부서, 현장, 프로젝트, 적요코드를 다른 코드로 변환 | 자금항목 데이터는 변환되지 않거나 제한될 수 있음 |
| 일괄적용 | 액션바 | 선택 전표에 구분, 거래처, 적요명 등을 일괄 반영 | 선택 범위와 조회 조건 확인 필요 |
| 전표라인삽입 | `Insert` | 같은 전표번호 안에 빈 라인을 삽입 | 삽입 후 대차균형 재확인 |
| 전표묶기 | `F7` | 여러 전표를 같은 번호로 묶거나 분리 | 같은 일자 전표 기준으로 처리 |
| 전표복사 | `F4` | 선택 전표를 다른 일자로 복사 | 자금항목, 증빙, 자동전표 연결 제한 확인 |
| 전표이동 | `Ctrl+M` | 선택 전표를 지정 일자로 이동 | 일마감, 자금항목, 자동전표 자료 제한 확인 |
| 체크펜 | 우클릭 메뉴 | 추후 확인할 전표를 표시 또는 해제 | 검토 상태와 실제 승인 상태를 혼동하지 않기 |
| 삭제 | `F5` 또는 액션바 | 선택 전표 삭제 | 관리항목, 마감, 승인 상태 확인 필요 |
| 인쇄 | `F9` 또는 액션바 | 선택 전표 출력 | 체크 범위와 출력 옵션 확인 |

## 코드변환

코드변환은 이미 입력된 전표의 코드값을 다른 코드값으로 바꾸는 기능이다. 대량 입력 후 거래처를 정리하거나, 계정과목 통폐합, 부서/현장/프로젝트 코드 변경, 적요코드 정비가 필요할 때 사용한다.

### 변환 가능한 대표 코드

| 코드 종류 | 사용 예 | 확인 포인트 |
|---|---|---|
| 거래처 | A 거래처로 입력된 전표를 B 거래처로 변경 | 거래처별 원장, 채권채무 잔액 영향 확인 |
| 계정과목 | 잘못 입력한 계정을 올바른 계정으로 변경 | 부가세, 관리항목, 결산 분류 영향 확인 |
| 사원 | 담당 사원 코드 변경 | 사원별 비용 또는 채권채무 관리 영향 확인 |
| 부서 | 부서 비용 귀속 변경 | 부서 손익과 예산 관리 영향 확인 |
| 현장 | 현장별 원가 귀속 변경 | 현장 원가와 손익 영향 확인 |
| 프로젝트 | 프로젝트 비용 또는 수익 귀속 변경 | 프로젝트 손익과 정산 영향 확인 |
| 적요코드 | 반복 적요 또는 세무 반영 적요 변경 | 의제매입 등 서식 반영 조건 영향 확인 |

### 코드변환 제한과 위험

WEHAGO 고객센터 문서 기준으로 자금항목 관련 데이터는 코드변환 대상에서 제외되거나 제한된다. 예를 들어 당좌, 받을어음, 지급어음, 차입금, 미불관리 같은 자금성 데이터는 단순 코드 치환으로 바꾸면 후속 관리대장의 정합성이 깨질 수 있다.

| 상황 | 위험 | 권장 확인 |
|---|---|---|
| 계정과목을 자금관리 계정으로 변경 | 관리항목이 새로 필요하지만 기존 전표에 정보가 없음 | 변경 후 관리항목 누락 여부 확인 |
| 자금관리 계정을 일반 계정으로 변경 | 어음/차입금 관리 데이터가 고립될 수 있음 | 후속 현황 메뉴 반영 여부 확인 |
| 적요코드를 세무 반영 적요로 변경 | 신고 부속서류에 새로 반영될 수 있음 | 신고서 관리용 탭 반영 여부 확인 |
| 거래처를 일괄 변경 | 거래처별 원장과 채권채무 잔액이 크게 바뀜 | 변경 전후 거래처별 잔액 비교 |
| 변경후 계정에 적요코드가 없음 | 적요코드가 공란 또는 0으로 바뀔 수 있음 | 세무서식 반영 누락 확인 |

BK 설계에서는 코드변환을 단순 대량 업데이트가 아니라 `bulk amendment` 성격으로 다뤄야 한다. 실행자, 대상 범위, 변경 전/후 값, 사유, 영향받은 전표 수를 감사로그에 남기고, 마감된 기간이나 승인된 전표는 별도 권한과 절차가 필요하다.

## 일괄적용

일괄적용은 같은 값을 여러 전표에 한 번에 넣는 기능이다. 예를 들어 같은 적요명을 여러 라인에 넣거나, 선택한 전표의 거래처나 구분을 한 번에 맞추는 데 사용한다.

| 적용 대상 | 사용 예 | 주의점 |
|---|---|---|
| 구분 | 여러 라인을 `3.차변` 또는 `4.대변`으로 맞춤 | 차대 방향이 바뀌므로 금액 위치와 대차균형 확인 |
| 거래처 | 누락된 거래처를 선택 라인에 일괄 입력 | 모든 선택 라인이 같은 거래처인지 확인 |
| 적요명 | 반복 거래의 설명을 통일 | 세무 반영 적요코드와 단순 적요명을 혼동하지 않기 |

실행 전에는 다음을 확인한다.

- 현재 조건검색이 걸려 있어 일부 전표만 보이는 상태인지 확인한다.
- 체크한 전표와 커서가 위치한 라인이 의도한 대상인지 확인한다.
- 일괄적용 대상이 회계 의미상 같은 거래인지 확인한다.
- 적용 후 전표 소계와 원장 조회 결과가 의도대로 바뀌었는지 확인한다.

## 전표라인 삽입

전표라인 삽입은 선택한 전표번호 안에 빈 라인을 추가하는 기능이다. 대체전표에서 차변 또는 대변 라인이 빠졌거나, 한 거래를 여러 계정으로 나누어 입력해야 할 때 사용한다.

| 사용 상황 | 처리 방향 |
|---|---|
| 상대 계정 라인이 빠짐 | 같은 전표번호에 빈 라인을 삽입하고 상대 계정과 금액 입력 |
| 비용을 여러 부서로 나눔 | 같은 전표번호 안에 라인을 추가해 부서별 금액 분할 |
| 세금/수수료/할인액을 별도 계정으로 분리 | 기존 전표번호에 보조 라인을 추가 |
| 입력 순서를 보정 | 필요한 위치 위쪽에 라인을 삽입하고 전표묶기 또는 순서 조정 기능과 함께 사용 |

삽입 후에는 전표번호 단위로 차변 합계와 대변 합계가 같은지 확인해야 한다. 라인을 삽입한 순간에는 빈 줄이므로 저장 전까지 계정, 거래처, 관리항목, 적요, 금액을 모두 채워야 한다.

## 전표묶기

전표묶기는 같은 일자의 여러 전표를 하나의 전표번호로 묶거나, 하나의 전표번호 안에 있던 라인을 분리하는 기능이다. Smart A 10에서는 전표번호를 임의로 직접 수정하기보다 전표묶기 기능으로 번호 구조를 조정하는 흐름을 사용한다.

### 묶기

| 단계 | 내용 |
|---:|---|
| 1 | 같은 일자에 있는 묶을 전표들을 체크한다. |
| 2 | `전표묶기(F7)`를 실행한다. |
| 3 | 묶기 대상을 확인하고 실행한다. |
| 4 | 결과적으로 같은 전표번호 안에 차변/대변 라인이 모였는지 확인한다. |
| 5 | 대차균형과 라인 순서를 다시 확인한다. |

### 분리

| 단계 | 내용 |
|---:|---|
| 1 | 이미 묶인 전표 중 분리할 라인을 체크한다. |
| 2 | 전표묶기 화면에서 새로운 전표 생성 옵션을 사용한다. |
| 3 | 분리된 라인이 새로운 전표번호를 받았는지 확인한다. |
| 4 | 기존 전표와 새 전표가 각각 대차균형을 이루는지 확인한다. |

전표묶기는 전표번호 구조를 바꾸므로 회계 검토 흐름에 영향이 있다. 특히 승인, 첨부, 증빙, 자금항목이 연결된 전표는 묶기와 분리 후에도 연결정보가 유지되는지 확인해야 한다.

## 전표복사와 전표이동

전표복사와 전표이동은 반복 거래 입력에는 매우 유용하지만, 원본 전표의 연결정보를 어떻게 처리할지에 따라 위험이 있다.

| 기능 | 의미 | 사용 예 | 주요 제한 |
|---|---|---|---|
| 전표복사 | 원본 전표를 유지하고 다른 일자에 같은 내용을 복제 | 매월 반복되는 임차료, 수수료, 내부 대체 | 자금항목, 증빙, 자동전표, 업무용승용차 등 연결정보 복제 주의 |
| 전표이동 | 원본 전표의 거래일자를 다른 일자로 옮김 | 일자를 잘못 입력한 전표 수정 | 일마감, 자금항목, 계좌관리, 증빙, 자동전표 자료 제한 |

### 전표복사 시 확인

- 복사 대상 일자가 맞는지 확인한다.
- 원본 거래처, 계정, 적요가 새 일자에도 그대로 맞는지 확인한다.
- 관리항목을 그대로 복사해도 되는지 확인한다.
- 어음번호, 차입금번호처럼 고유 식별자는 복사하면 안 되는지 확인한다.
- 복사 후 금액과 적요를 새 거래에 맞게 수정한다.

### 전표이동 시 확인

- 이동 전후의 회계기간이 같은지 확인한다.
- 이동 후 마감 기간에 들어가거나 마감 기간에서 빠져나오는지 확인한다.
- 부가세 신고기간, 월마감, 결산, 보고 패키지에 영향이 있는지 확인한다.
- 자금관리 현황의 만기, 수취, 지급일과 전표일자의 관계가 깨지지 않는지 확인한다.

저장소 SmartA 분석 문서에서도 자금항목, 일마감, 계좌관리, 증빙, 자동전표 관련 데이터는 이동 또는 복사에 제한이 있음을 별도로 기록하고 있다. BK에서는 전표복사와 전표이동을 모두 감사 가능한 변경 이벤트로 모델링해야 한다.

## 체크펜

체크펜은 전표를 나중에 다시 확인하기 위해 표시하는 기능이다. 회계적으로 승인되었다는 뜻이 아니라, 검토 대상 또는 확인 필요 표시로 이해하는 것이 안전하다.

| 사용 상황 | 설명 |
|---|---|
| 금액 확인 필요 | 증빙과 금액이 맞는지 나중에 다시 확인할 전표 표시 |
| 거래처 확인 필요 | 거래처 코드가 맞는지 검토할 전표 표시 |
| 관리항목 누락 의심 | 부서, 현장, 프로젝트, 차량, 자금항목 누락 후보 표시 |
| 담당자 확인 요청 | 다른 담당자에게 확인이 필요한 전표 표시 |
| 신고 전 점검 | 부가세, 결산, 보고 전 검토 목록으로 활용 |

체크펜은 우클릭 메뉴 등에서 설정/해제할 수 있다. 검토가 끝난 뒤에는 표시를 해제해 남아 있는 검토 대상을 명확히 해야 한다. BK에서는 체크펜을 `reviewFlag`, `reviewReason`, `reviewedBy`, `reviewedAt` 같은 검토 메타로 확장할 수 있다.

## 자금항목과 액션바 기능 제한 정리

| 기능 | 일반 전표 | 자금항목 전표 | 설계상 권장 처리 |
|---|---|---|---|
| 삭제 | 임시저장 또는 미승인 상태에서는 비교적 단순 삭제 가능 | 어음/차입금/미불관리 연결자료 확인 필요 | 연결 보조원장과 함께 검증, 승인 후에는 취소/역분개 |
| 코드변환 | 거래처, 계정, 부서 등 일괄 변경 가능 | 자금항목 데이터는 변환 제외 또는 제한 | 변환 전후 영향분석과 감사로그 |
| 일괄적용 | 구분, 거래처, 적요 등 반복 입력 가능 | 관리항목 의미가 달라질 수 있음 | 선택 범위 검증, 자금항목 직접 일괄변경 제한 |
| 전표라인삽입 | 누락 라인 보완 가능 | 자금항목 라인 추가 시 보조정보 필수 | 저장 전 관리항목 필수 검증 |
| 전표묶기 | 같은 일자 전표 묶기/분리 가능 | 연결정보 유지 여부 확인 필요 | 묶기/분리 이벤트 이력화 |
| 전표복사 | 반복 거래 복제 가능 | 어음번호/차입금번호 같은 고유값 복제 위험 | 고유 보조정보는 복사 제외 또는 재입력 |
| 전표이동 | 일자 오입력 수정 가능 | 만기/상환/증빙일자와 충돌 가능 | 기간 잠금, 신고기간 영향, 보조원장 정합성 검증 |
| 체크펜 | 검토 표시 가능 | 자금관리 검토 대상으로 유용 | 사유와 담당자까지 남기면 좋음 |

## 입력 전 체크리스트

- 받을어음, 지급어음, 차입금 중 어떤 자금항목인지 먼저 구분한다.
- 해당 계정과목에 필요한 관리항목이 설정되어 있는지 확인한다.
- 지급어음은 어음집계표 또는 어음등록에서 사용할 어음이 사전에 등록되어 있는지 확인한다.
- 받을어음은 어음번호, 만기일, 지급은행, 거래처 정보를 준비한다.
- 차입금은 차입처, 차입기간, 원금, 상환방식, 이자율, 자금용도를 준비한다.
- 거래일자와 증빙일자, 만기일, 상환일이 서로 다른 경우 어느 날짜를 전표일자로 쓸지 확인한다.
- 일마감 또는 월마감 기간에 입력하려는 전표가 아닌지 확인한다.

## 입력 후 검토 체크리스트

- 전표번호 단위로 차변 합계와 대변 합계가 일치하는지 확인한다.
- 받을어음현황에서 만기일별, 지급은행별, 거래처별 조회가 의도대로 되는지 확인한다.
- 지급어음현황과 어음집계표에서 등록 어음과 전표가 연결됐는지 확인한다.
- 차입금스케줄관리에서 상환스케줄과 잔액이 맞는지 확인한다.
- 관리항목이 붙은 전표를 삭제, 복사, 이동, 코드변환한 경우 후속 현황 메뉴를 다시 조회한다.
- 체크펜으로 표시한 전표는 검토 후 해제하거나 사유를 정리한다.
- 마감 전에는 자금항목 누락, 어음번호 중복, 차입금번호 중복, 만기일 오류를 별도로 점검한다.

## BK 설계 반영 메모

이 문서는 위하고 사용법 정리이며 BK의 공식 스펙은 아니다. 다만 현재 `kat_bk` 설계와 연결할 때는 아래 포인트를 참고할 수 있다.

- BK의 `journals` 도메인은 자금성 보조항목을 전표 라인의 단순 문자열이 아니라 별도 보조원장 참조로 모델링해야 한다.
- 받을어음과 지급어음은 `funds-ar` 또는 별도 note 관리 aggregate와 연결하고, 어음번호, 만기일, 상태 전환 이력을 관리해야 한다.
- 지급어음은 전표 입력 전에 어음번호의 사전 등록과 사용 가능 상태 검증이 필요하다.
- 차입금은 전표 라인과 차입금 스케줄을 함께 생성하거나 연결해야 하며, 원금 상환과 이자 지급은 별도 이벤트로 추적해야 한다.
- 자금항목이 연결된 전표는 코드변환, 전표복사, 전표이동에서 기본적으로 제한하거나, 연결 데이터 재검증 절차를 거쳐야 한다.
- 전표묶기와 전표분리는 전표번호만 바꾸는 UI 기능처럼 보여도 실제로는 전표 구조 변경이므로 감사로그 대상이다.
- 체크펜은 단순 표시 기능을 넘어 검토 플래그, 검토 사유, 담당자, 완료일시로 확장할 수 있다.
- 일마감 또는 월마감 이후에는 삭제와 이동을 막고, 필요한 경우 마감해제 승인, 취소전표, 역분개 같은 통제 흐름을 사용해야 한다.
- 코드변환과 일괄적용은 대량 변경 기능이므로 변경 전후 스냅샷, 대상 건수, 실행자, 사유를 기록해야 한다.
- 어음, 차입금, 카드계좌, 업무용승용차처럼 후속 장부가 있는 관리항목은 API DTO에서도 별도 타입으로 구분해 누락 검증을 해야 한다.
