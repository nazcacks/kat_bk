| 구분 | 채택 기술 | 비고 |
|---|---|---|

| 프론트엔드 | **React 18 + TypeScript** | 운영 콘솔(OP) / 업무 화면(TN) 2개 SPA, 공통 디자인시스템 패키지 공유 |

### 0.1 기술 스택 적용 원칙

- **모듈러 모놀리식**: NestJS Module = 상세설계서 도메인 서비스(13.1) 1:1 대응. 도메인 간 호출은 인터페이스(포트)를 통하며, 직접 테이블 접근 금지(테넌트 경계·SOD 보장).
- **멀티테넌시 강제**: 모든 요청은 `TenantContext`(미들웨어·`AsyncLocalStorage`)에서 `tenant_id`·접근모드(`VIEW`/`SUPPORT_SESSION`/`BOOKKEEPING`/`BREAK_GLASS`)·기장모드를 주입한다. DB 커넥션마다 `SET app.current_tenant` → RLS 정책 평가(애플리케이션+DB 이중 안전장치, 상세 2장).
- **회계 불변성**: 회계 핵심 데이터 물리 삭제 금지, append-only 이력·감사필드(`created_by`/`version`/`source`) 공통 적용(상세 12.3).
- **2개 프론트 분리**: 운영 콘솔과 업무 화면은 도메인·세션을 분리(상세 3.4)하되, 공통 컴포넌트(대량 그리드·Step-up 모달·마스킹·상태 배지)는 공유 패키지로 관리(화면설계서 2장).

## 1. 전체 아키텍처 개요

```
┌────────────────────┐      ┌────────────────────┐
│  운영 콘솔 SPA (OP) │      │  업무 화면 SPA (TN) │   React 18
└─────────┬──────────┘      └─────────┬──────────┘
          │  HTTPS / JWT(세션 분리)    │
┌─────────┴───────────────────────────┴──────────┐
│            API Gateway / BFF (NestJS)            │
│  TenantContext · AuthGuard · RoleGuard · 마감잠금 │
│  인터셉터 · 멱등키 · RateLimit · 감사로그 인터셉터  │
├──────────────────────────────────────────────────┤
│  도메인 모듈 (NestJS Modules = 상세 13.1 서비스)    │
│  Auth / Tenant / Delegation / Standard / Billing  │
│  Journal / Ledger / Closing / Tax / Treasury …    │
│  Bookkeeping(21장) / ReportPackage(11.13) …       │
├──────────────────────────────────────────────────┤
│  공통 기반: Notification·Audit·KMS·FeatureFlag    │
├───────────────┬───────────────┬──────────────────┤
│  PostgreSQL16 │   Redis        │   Queue(BullMQ)  │
│  (RLS·파티션) │ (캐시·Redlock) │  (ETL·리포트·알림)│
└───────────────┴───────────────┴──────────────────┘
        │ 비동기 워커 (ETL·스크래핑·리포트·배치·이상탐지)
        └─→ S3(증빙·스냅샷·Export) · KMS · 외부연계(NTS·은행·SSO IdP)
```

- 단계와 무관하게 **공통 기반 레이어**(2절)는 Phase 1부터 골격을 구축하고 단계별로 확장한다(상세 22장 「공통」).

## 6. 데이터베이스 전략 (PostgreSQL)

- **멀티테넌시 3티어**: `SHARED`(공유 스키마+`tenant_id`+RLS, 기본·P1) → `SCHEMA`(전용 스키마) → `DEDICATED`(전용 DB)는 P2에서 전환 잡으로 지원. 전환은 마감 후 점검 윈도우 기반 반자동 이행 표준, 실시간 전환 시 CDC·아웃박스·쓰기 버퍼링(2.1.1).
- **RLS**: 모든 테넌트 테이블에 `USING (tenant_id = current_setting('app.current_tenant')::uuid)` 정책. 운영자 관리자 접근은 별도 정책+로그.
- **파티셔닝**: 전표라인·원장·접근로그·변경로그·연계로그는 `tenant_id`+회계기간/일자 기준 파티셔닝(12.3).
- **불변성·이력**: 회계 핵심 데이터 물리 삭제 금지, append-only 로그·이력 테이블, 감사필드(`created_at/by`·`version`·`source`).
- **멱등·채번**: 외부 수신·자동전표·전송 API는 (`tenant_id`, 업무키) unique + `idempotency_key`·`request_hash`.
- **민감정보**: 주민번호·계좌·카드·급여는 `pgcrypto` 암호화 컬럼 + 마스킹 정책 + 평문조회 로그 FK(P2).
- **마이그레이션 거버넌스**: 스키마 변경은 코드 리뷰+무중단(Expand-Contract) 패턴. RLS·파티션·복합 FK는 raw SQL 마이그레이션으로 버전 관리.

## 7. 마이그레이션 · 온보딩 (전 단계 공통)

- **신규 온보딩**(P1~): 구독 개시 → 기장 모드 설정·양측 동의(21장, P2) → 차원 설정·표준 채택 → 계정/거래처 이관 → **개시잔액 시산표 균형 검증**.
- **기장 이관 온보딩**(22.1, P3/N항목): 전임 COA↔BK 표준 COA 매핑(다국어) → 전임 GL/TB 수신·개시잔액 적재 + 시산표 tie-out(11.14) → 과거 분개 선택적 적재 → 미결항목·증빙 인수. tie-out 전건 통과 후 정상 기장 개시.
- 차원 일괄배정은 전용 도구로 제공, 마감기간 소급 변경은 별도 승인·감사로그.

### 9.6 크리티컬 패스 · 선후행 의존

```
공통기반(RLS·TenantContext, M1-3)
   └─→ 전표 코어(M4-5) ─→ 장부(M5-6) ─→ 마감 워크플로우+게이트 훅(M6-7)   ★ Phase 1 임계경로
          │
          └─→ 기장 모드·워크벤치(21장, M10-13)   ★ Phase 2 임계경로 (전표·결재 의존)
                  └─→ Ack·협업(M12-13)
                          │
                          └─→ 보고패키지·tie-out·미결항목·원가(M15-19)
                                  └─→ 마감 게이트 조건 충족(M19) ─→ GA(M20)   ★ Phase 3 임계경로
```

- **임계경로 핵심**: `RLS/TenantContext → 전표 코어 → 마감 워크플로우`. Phase 1에서 마감 게이트 **훅을 미리 심고**(4절), Phase 3에서 조건을 채우는 구조라 P1 마감 설계 지연이 전 단계로 전파된다.
- **선행 골격**: `OpenItemService`(미결항목)는 P2에서 Ack 연계용 골격을 먼저 만들고 P3에서 차단 로직을 완성한다(부록 A).
- **병행 가능**: 운영 콘솔(OP)·증빙·차원·표준은 전표 코어와 병행 개발 가능. 세무·자금·과금은 Phase 2 내에서 상호 독립적으로 병행한다.

### 9.7 리스크 · 버퍼 · 점진 배포

- 각 Phase 마지막 스프린트(S14/S12/S14)는 **안정화·회귀·침투 테스트 버퍼**로 고정한다.
- 각 Phase 종료 시 **회귀 테스트 + RLS/권한/마감 게이트 침투 테스트 + 화면-API-엔티티 추적성 점검**(화면설계서 8.3)을 게이트로 수행한다.
- 선결 정책(8절) 미확정은 최대 일정 리스크 — 해당 Phase 착수 1스프린트 전까지 확정하지 못하면 일정 재조정한다.
- v4.0 신설 화면(TN-22~31 등)은 기능 플래그(`feature.*`)·업종(`industryType`)으로 다크 런치 후 단계 활성화하여 릴리스 리스크를 분산한다.
