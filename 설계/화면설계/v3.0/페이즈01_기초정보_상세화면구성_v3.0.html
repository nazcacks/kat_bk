1. 문서 개요 · 페이즈 01 범위

목적페이즈 01 구현 착수를 위해 기초정보·조직·표준 설정 화면별 레이아웃(와이어프레임)·입력 필드·액션/검증·연계 API·권한·상태를 구현 가능한 수준으로 정의한다. 플랫폼 기반(테넌트/인증/권한/표준)과 BK 기초정보(회사·환경·조직·거래처·계정과목·차원 마스터)를 하나의 페이즈로 묶어, 후행 전표·결산·신고 화면의 부모 엔티티를 모두 확정한다.
페이즈 정의멀티테넌시(SHARED+RLS)·인증/2FA·운영 콘솔(테넌트/구독/표준)·권한 위임/가드레일·관리자 접근 로그·공통 표준 배포 + 회사등록·환경설정·6 관리차원 마스터(부서/사원/거래처/계정과목/현장/프로젝트)·차량·외주처·코드변환·마감후이월. BK의 "기수 단일파일" 모델을 tenant + fiscal_period 모델로 일반화한다.
화면 구성플랫폼/기초 BF-00~BF-12 13화면(테넌트 스위처·운영조직·테넌트 콘솔·기장 워크플로우·보안 사용자·인증 정책·접근 감사·권한 매트릭스·표준 카탈로그·차원/조직·구독/청구·알림·메뉴 발행) + BK 기초정보 SA-BAS-01~SA-BAS-14 14화면 = 총 27화면.
표기 원칙화면 ID는 본 문서 전용으로 BF-xx(플랫폼/기초)·SA-BAS-xx(BK 기초정보)를 사용하고, 설계서 bk_설계서_v2.1.md 0.2·25.1 절과 연결한다. 검증은 경고(저장가능)·차단(저장불가) 2단계로 표기한다.
본 페이즈는 설정·마스터 중심이므로, 화면 대부분이 좌측 목록/트리 + 중앙 상세 입력(탭) + 우측 정책/검증 패널의 3분할이거나 매트릭스·콘솔·배치 형태다. 입력 마스터 화면은 v2.1 전표 화면의 핵심 그리드 + 우측 패널 패턴을 응용해, 코드·명칭 등 필수만 그리드에 두고 차원·정책·연계는 우측 패널/탭으로 분리했다.
페이즈 01 연계 핵심 — SA-BAS-03 환경설정의 관리코드 사용여부와 SA-BAS-07 계정과목별 관리항목 설정이 전표(P03) 우측 「선택 라인 추가입력」 패널의 노출/필수를 결정한다. 즉 본 페이즈에서 어떤 관리차원(사업장·코스트센터·부서·프로젝트·현장·사원)을 켜고, 어떤 계정에 관리항목을 붙이느냐가 이후 모든 전표 입력 UX를 좌우한다.

2. 공통 UI 표준 (요약)

2.1 화면 셸 · 마스터 입력 표준

  • 셸: 상단 GNB(회사명·기장 모드 배지·알림·사용자) + 좌측 LNB(기초정보/전표/장부·결산/설정·운영) + 콘텐츠. 메뉴는 Role 메뉴권한(BF-12), 버튼은 행위권한(BF-07)으로 렌더링.
  • 마스터 3분할: 좌측 목록/조직 트리 + 중앙 상세 탭 입력 + 우측 정책·검증·연계 패널. 코드도움(F2), 다음 행(Enter), 저장(Ctrl+S).
  • 2단계 검증: Amber 경고(저장 가능) / Red 차단(저장 불가). 서버 검증 이중 유지.
  • 마스킹(17장): 계좌·카드·주민번호 기본 마스킹, 👁 평문 조회는 Step-up 인증 + 사유 + 자동 재마스킹.
  • 변경 이력: 모든 마스터는 append-only 변경이력, 물리 삭제 금지. 사용여부 토글·유효일자로 논리 비활성(12.3).
  • RLS 스코프: 회사권한(전체회사/관리회사)이 테넌트 행 수준 접근 범위를 결정(SA-BAS-02). 모든 마스터 조회·저장은 tenant_id RLS 필터를 통과.
  • 6 관리차원: 사업장(*기본필수)·코스트센터·부서·프로젝트·현장·사원. DimensionConfig.required로 회사별 필수/선택 조정(BF-09·SA-BAS-03·SA-BAS-04~09).
