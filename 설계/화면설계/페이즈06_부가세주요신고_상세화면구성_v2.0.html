1. 문서 개요 · 페이즈 06 범위

목적페이즈 06 구현 착수를 위해 부가가치세 주요신고·기초 신고서류 화면별 레이아웃(와이어프레임)·입력 필드·액션/검증·연계 API·권한·마감상태를 구현 가능한 수준으로 정의한다. 매입매출전표(P03)의 과세유형 코드(11~64)별 자동 집계를 부가세 신고서식으로 변환하는 신고 엔진과, 합계표·명세서·전자신고·납부서까지의 작업 흐름을 하나의 페이즈로 묶는다.
페이즈 정의부가가치세 신고서(일반/간이) · 세무리스크 자동검증(MRI) · 매출/매입처별 세금계산서합계표 · 신용카드매출전표등 수령명세서·발행집계표 · 계산서합계표 · 전자신고(홈택스 변환·전송) · 납부서(2D바코드) · 부동산임대공급가액·건물관리명세서 · 의제매입/재활용폐자원/구리스크랩 매입세액공제 · 현금매출·동물진료용역 매출명세서. 신고기간은 예정(1·3분기)·확정(반기)·영세율조기환급·기한후를 구분하고 수정신고 차수로 이력 관리한다.
화면 구성부가세 신고서·검증 SA-VAT-01~06 6화면 + 전자신고·납부 SA-VAT-07~08 2화면 + 첨부·공제 명세 SA-VAT-09~15 7화면 = 총 15화면.
표기 원칙화면 ID는 본 문서 전용으로 SA-VAT-xx를 사용하고, 설계서 bk_설계서_v2.1.md 25.11·25.12 절과 연결한다. 검증은 경고(저장가능)·차단(저장불가) 2단계, 집계 정합은 정상으로 표기한다.
본 페이즈 화면 대부분은 신고서 작업대(workbench) 구조다 — 좌측 신고조건 패널(기수·신고구분·과세단위·수정차수) + 중앙 탭형 신고서 본문(기본/명세/집계/검증/전자신고) + 우측 연계·산출물 패널(MRI·납부서·전자신고 파일). 합계표·명세서류는 좌측 마스터 그리드 + 우측 상세 입력 패널의 master-detail 구조를 사용한다.
페이즈 06 연계 핵심 — 모든 신고서는 P03 매입매출전표의 세무구분 코드를 원천으로 RLS 격리된 테넌트별 부가세 집계 뷰(vat_aggregate_view)에서 집계된다. 따라서 전표 마감(P03)이 선행되어야 신고서 작성이 가능하며, 신고서가 저장·마감되면 역방향 전표 수정이 차단된다. 정기신고 마감 후에만 동일기간 수정신고(차수+1)를 작성할 수 있다.

2. 공통 UI 표준 (요약)

2.1 신고 화면 셸 · 작업대 표준

  • 셸: 상단 GNB(회사명·기장 모드 배지·신고기간 컨텍스트·사용자) + 좌측 LNB(부가세/신고 그룹) + 콘텐츠. 메뉴는 Role 메뉴권한(BF-12), 버튼은 행위권한(BF-07)으로 렌더링.
  • 신고 작업대 3분할: 좌측 신고조건(기수·신고구분·과세단위·수정차수) + 중앙 탭형 신고서 본문 + 우측 연계/산출물 패널. 불러오기(F8), MRI 검증(F9), 저장(Ctrl+S).
  • 과세단위 키: 사업장별 / 총괄납부 / 사업자단위과세 중 선택. 신고서·합계표·납부서가 동일 과세단위 키로 일관 집계되어야 한다.
  • 2단계 검증: Amber 경고(저장 가능) / Red 차단(저장 불가) / 정상(집계 정합). 클라이언트·서버 이중 검증.
  • 마감 가드: P03 전표 마감(TEMP_CLOSED/CLOSED/LOCKED) 상태에서만 신고서 작성, 신고서 마감 시 원천 전표 잠금. 모든 변경은 append-only 감사로그, 물리 삭제 금지.
  • 마스킹: 주민등록번호·사업자등록번호 일부 마스킹 기본, 👁 평문 조회는 Step-up 인증 + 사유 + 자동 재마스킹(17장).
  • RLS 스코프: 모든 신고서 집계·조회·저장은 tenant_id RLS 필터 통과. 전자(세금)계산서는 국세청 연동 데이터를 테넌트 단위로 격리 매핑.
