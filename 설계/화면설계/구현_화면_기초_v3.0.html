1. 문서 개요

목적BK 서비스의 기초·기준정보 계층을 운영·관리하는 화면을 정의한다. 각 엔티티의 모든 필드(세부항목)를 목록/상세 폼·그리드·필드 매핑표로 빠짐없이 표현하여 구현 착수의 기준으로 삼는다.
범위관리회사(운영조직)·이용회사(테넌트)·기장 관계 설정·사용자·인증 정책·관리자 접근 거버넌스·권한/권한위임·공통코드/공통표준·차원(관리항목)·구독/요금·알림. 전표·마감·신고 등 업무 트랜잭션 화면은 구현_화면설계_v1.0.html에서 다룬다.
화면 구성 원칙도메인별로 ① 목록(List) + 검색·필터, ② 상세/편집 폼(Detail)(전 필드), ③ 하위 엔티티 탭/그리드, ④ 필드 매핑표(엔티티·타입·필수·제약), ⑤ 액션·검증·권한을 함께 제시한다.
마스킹·암호화사업자/법인등록번호·연락처·계좌·시크릿 등 enc/mask 필드는 기본 마스킹, 👁 평문 조회는 Step-up 인증 + 사유 기록 + 자동 재마스킹(상세 17·20.1).
금지(보안)비밀번호·결제수단(카드·계좌번호) 평문 입력 대행 금지. 본 화면은 토큰/PG·KMS 참조만 저장하고, 자격증명 입력은 사용자가 직접 수행한다.

2. 공통 UI · 표준

2.1 화면 셸·공통 컴포넌트

  • 셸: 상단 GNB(조직/회사·환경 배지·알림·사용자) + 좌측 LNB(도메인 메뉴) + 콘텐츠. 메뉴는 Role 메뉴권한, 버튼은 행위권한으로 렌더링(상세 4.3).
  • 목록 표준: 검색바 + 필터 + 정렬 + 페이지네이션(대량은 가상 스크롤), 행 클릭 → 상세 슬라이드/페이지. 상태는 배지로 표시.
  • 폼 표준: 필수 *, 인라인 검증(Amber 경고 / Red 차단), 저장 전 변경요약, 저장 시 version 낙관적 잠금.
  • 이력·감사: 모든 상세에 "변경이력" 탭(DataChangeLog append-only·해시체인, 12.3). 물리 삭제 금지 — 비활성/상태전이로 처리.

8. 메뉴 · 권한 관리

BF-12메뉴 · 권한 관리 OPTN

전체 서비스 메뉴(GNB·LNB·탭·액션)를 Menu 마스터로 카탈로그화하고, 각 메뉴의 노출·진입·버튼(행위)을 역할(Role) 기준으로 권한 매핑하는 화면이다. 상세설계 §4.3 RBAC를 화면화한 것으로, Role = 메뉴권한(어떤 메뉴가 보이는가) + 행위권한(조회/등록/수정/삭제/승인/반려/마감/전송/출력) + 데이터범위 + 민감정보권한의 조합이다. 메뉴 노출은 Permission(category=MENU), 버튼 렌더링은 Permission(category=ACTION)으로 평가된다.

핵심 개념 — ① Menu: 메뉴 정의(코드·계층·채널·화면 연결·필요 권한). ② RoleMenuPermission: 역할 × 메뉴(노출/진입). ③ MenuActionGrant: 역할 × 메뉴 × 행위(버튼 단위). ④ MenuVersion: 메뉴 구성 발행/롤백(스냅샷 WORM). 메뉴는 물리 삭제 금지 — active=false(숨김)·버전 보존으로 처리(부록 A).

부록 C. 코드값(Enum) 집합

관리 방식 — 이 표는 "시스템 고정 Enum"이다(업무 공통코드와 구분). 아래 값들은 도메인 모델·상태머신·보안 로직에 결합된 구조적 타입으로, 애플리케이션 코드(TS enum/상수)와 데이터베이스(PostgreSQL enum 타입 또는 CHECK 제약 + 필요 시 lookup 테이블)에 동시에 선언한다. 단일 출처는 본 부록(상세 15장 Enum Master)이며, 런타임에 사용자가 추가·수정할 수 없고 변경은 코드 배포 + DB 마이그레이션으로만 한다(값 추가는 append 전용, rename/재사용 금지, 폐기는 deprecated 처리로 하위호환·과거데이터 보존). 표시명·다국어 라벨은 코드값과 분리해 메시지 카탈로그/STD_ACCOUNT_L10N으로 관리한다.
