1. 문서 개요 · 페이즈 10 범위

목적페이즈 10 구현 착수를 위해 데이터관리·운영(데이터 백업/복구·내보내기/올리기·삭제전표 복구·기수변경) 화면별 레이아웃(와이어프레임)·조건 필드·실행 콘솔·검증·연계 API·권한·상태를 구현 가능한 수준으로 정의한다. 레거시 Smart A의 단일 .wdb/.whg/.spwdata 파일 기반 백업·이관 모델을 BK의 멀티테넌트(SHARED+RLS)·회계기간(fiscal_period)·비동기 잡(JobRun) 모델로 일반화한다.
페이즈 정의운영 도구 7화면: 소프트삭제 전표 복구(휴지통)·기수별 백업/복구·SmartA 호환 익스포트(기수·월/분기)·SmartA 백업 임포트(전체기수·월/분기)·기수(회계기간) 정정. 모든 작업은 사전검증 → 잠금 확인 → 복구지점 생성 → 비동기 실행 → 결과/로그 4단계 콘솔로 표준화한다.
화면 구성데이터관리·운영 SA-DAT-01~SA-DAT-07 7화면 = 총 7화면. 화면 구조는 좌측 작업조건 패널 + 중앙 실행 콘솔(진행률·잡 그리드) + 우측 안전장치/검증 패널의 3분할 작업 콘솔로 통일한다.
표기 원칙화면 ID는 본 문서 전용으로 SA-DAT-xx를 사용하고, 설계서 bk_설계서_v2.1.md 25.20 절(데이터관리)과 연결한다. 검증은 경고(저장가능)·차단(저장불가) 2단계로 표기한다.
관련 페이즈P01(테넌트·권한 BF-07/BF-12·거래처/부서/사원 마스터)·P02(전기이월·초기 데이터 이관)·P03(전표 채번·소프트삭제 휴지통)·P05(마감·이월 상태). SA-DAT는 이들 페이즈가 만든 데이터의 백업·이관·정정·복구 운영 계층이다.
본 페이즈는 운영 콘솔 중심이므로 모든 화면이 동일한 작업 콘솔 3분할(작업조건·실행 콘솔·안전장치)을 공유한다. 진행률 바, 잡 큐 그리드, 복구지점, 체크섬, 비밀번호/면책 동의가 화면마다 반복 노출된다. 파괴적 작업(전면 재구성·기수변경·기간 덮어쓰기) 전에는 자동 스냅샷(복구지점)을 강제한다.
페이즈 10 연계 핵심 — 데이터 올리기/재구성(SA-DAT-05/06)은 기간/서식 단위로 기존 데이터를 삭제 후 덮어쓰기한다. 따라서 마감 상태(TEMP_CLOSED/CLOSED/LOCKED) 기간은 잠금 가드로 차단하고, 실행 전 복구지점(스냅샷)을 반드시 생성한다. 모든 작업 이력은 append-only 감사로그로 남기며 물리삭제는 금지한다.

2. 공통 UI 표준 (요약)

2.1 작업 콘솔 · 운영 화면 표준

  • 셸: 상단 GNB(회사명·기장 모드 배지·알림·사용자) + 좌측 LNB(설정·운영) + 콘텐츠. 운영 메뉴는 Role 메뉴권한(BF-12), 실행 버튼은 행위권한(BF-07)으로 렌더링.
  • 작업 콘솔 3분할: 좌측 작업조건(회사·기수·기간·옵션) + 중앙 실행 콘솔(단계 진행률·잡 큐 그리드·로그) + 우측 안전장치(테넌트 스코프·기수 잠금·체크섬·복구지점).
  • 4단계 파이프라인: 사전검증 → 잠금 확인 → 복구지점 생성 → 실행 → 결과/로그 다운로드. 각 단계는 서버 잡(JobRun)으로 비동기 실행, 폴링으로 진행률 갱신.
  • 2단계 검증: Amber 경고(저장/실행 가능) / Red 차단(실행 불가). 서버 검증 이중 유지.
  • 마스킹(17장): 계좌·카드·주민번호 기본 마스킹. 백업/복구 시 평문 추출은 Step-up 인증 + 사유 + 자동 재마스킹. 백업파일은 AES 암호화(비밀번호 1~8자리), 분실 시 복구 불가 면책 고지.
  • 변경/작업 이력: 모든 운영 작업은 append-only 작업 이력(작업ID·작업자·작업시간·파일명·건수·체크섬), 물리 삭제 금지(감사로그).
  • RLS 스코프: 모든 백업/복구/이관 잡은 tenant_id RLS 필터를 통과하며, 복구 대상 테넌트의 사업자번호·기수가 백업 무결성 키와 일치해야 진행한다.
  • 비동기 잡: 대량 처리는 PostgreSQL 16 파티션 단위 배치 + 백그라운드 워커. 실패 시 복구지점으로 자동 롤백, 부분 적재 방지.
