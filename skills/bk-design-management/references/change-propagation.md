# 설계 변경 전파 절차

최신 `설계/bk_설계서_v*.md` 또는 관련 설계 산출물이 바뀔 때, 같은 변경이 필요한 모든 연관 설계를 찾아 반영하기 위한 절차다. 설계 변경 요청, 리뷰 반영, 화면설계 역반영, DB/API/배치/테스트 산출물 정합화에 사용한다.

## 전파 원칙

- 최신 최종 설계서 24장의 후속 산출물 목록을 기본 전파 범위로 삼는다.
- 변경이 업무 규칙이면 도메인 장과 데이터/API/검증/테스트 영향까지 확인한다.
- 변경이 화면 동작이면 화면설계서, 메뉴구조도, 최신 최종 설계서 0.2, 11장, 12-15장 영향까지 확인한다.
- 변경이 데이터 구조이면 최신 최종 설계서 12장만 수정하지 말고 API, 검증, 배치, 마이그레이션, 테스트 영향까지 확인한다.
- 변경이 보안/권한/테넌트 격리이면 2장, 3-5장, 17-20장, 테스트 시나리오를 함께 확인한다.
- 영향을 받지 않는 산출물도 검토 결과에 "영향 없음"으로 남긴다.

## 전파 작업 순서

1. 원천 변경을 한 문장으로 정의한다.
2. 변경 유형을 산출물별 전파 매트릭스에 매핑한다.
3. 최신 최종 설계서 내부에서 0장, 소유 도메인 장, 12-15장, 17-21장, 23-24장을 확인한다.
4. 관련 외부 산출물 파일을 검색하고 필요한 부분만 읽는다.
5. 필요한 산출물을 모두 수정한다.
6. 변경 후 `rg -n`으로 핵심 용어, 화면 ID, 엔티티명, API명, 상태값이 중복/불일치 없이 반영됐는지 확인한다.
7. 최종 결과에 수정한 산출물과 "검토했지만 영향 없음" 산출물을 나눠 적는다.

## 전파 완료 조건

아래 항목이 모두 충족되어야 완료로 본다.

- 최신 최종 설계서의 원천 변경 장과 연관 장이 일치한다.
- 화면 변경이면 화면설계/메뉴구조도와 최신 최종 설계서 0.2/24장이 일치한다.
- DB/API/배치/검증 변경이면 12-15장과 24장이 일치한다.
- 보안/권한/테넌트 변경이면 2-5장, 17-20장, 테스트 영향이 일치한다.
- 미확정 값은 23장에 남아 있고 확정 규칙처럼 작성되지 않았다.
- 최종 응답에 전파 범위, 수정 파일, 영향 없음 판단, 남은 미확정 항목이 포함된다.
