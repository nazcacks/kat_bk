# BK 설계 맵

이 참고자료는 최신 `설계/bk_설계서_v*.md`에서 요청과 관련된 장을 빠르게 찾고, BK 회계/세무 서비스 구현 작업의 추적성을 유지하기 위해 사용한다.

## 현재 기준 파일

- 현재 최종 설계서: 명시 버전이 없으면 `설계/bk_설계서_v*.md` 중 가장 높은 버전. 2026-06-27 점검 기준 현재 기준은 `설계/bk_설계서_v3.1.md`
- 저장소에 있는 화면/원본/과거 설계 산출물: `설계/화면설계/*`, `설계/원본/*`, `설계/참조/*`
- 사용자가 특정 버전 변경이나 비교를 요청하지 않으면 과거 버전과 원천 문서는 비교용으로만 사용한다.
- v3.1의 핵심 추가 기준: `구현_매입매출_전표_v1.0.html`의 매입매출전표입력(`SA-JNL-02`/`WPV-01`) 화면설계가 §25.4.2에 전수 통합되었다. 매입매출/부가세 입력 작업은 §25.4.2를 먼저 보고 11.4, 12-15장으로 내려간다.

저장소 루트에서 유용한 명령:

```powershell
rg --files -g "bk_설계서_v*.md" "설계"
$currentDesign = Get-ChildItem -LiteralPath "설계" -Filter "bk_설계서_v*.md" | Sort-Object { [version]($_.BaseName -replace '^bk_설계서_v','') } -Descending | Select-Object -First 1
rg -n "^#{1,6}\s" $currentDesign.FullName
rg -n "전표|차원|API|검증|마감|기장|보조부" $currentDesign.FullName
Get-Content -Encoding UTF8 -LiteralPath $currentDesign.FullName -TotalCount 120
```

## 코드에 자주 영향을 주는 최신 설계 규칙

- `JV-01`이 기본 전표 입력 화면이다. 별도 입금/출금 메뉴를 중심 설계로 보지 않는다.
- `SA-JNL-02`는 매입매출전표 입력 화면이다. v3.1 기준 상단 세무헤더와 하단 분개라인의 2계층 구조이며, 화면 구현 기준은 `WPV-01`/`구현_매입매출_전표_v1.0.html`이다.
- 전표번호는 전표일자별로 매일 리셋된다. 외부 표기는 `YYYY-MM-DD-NNN`이며 키에는 `tenant_id`, `전표일자`, `전표번호`가 포함된다.
- 매입매출전표는 과세유형 코드(매출 11~25, 매입 51~64)를 단일 진실원으로 사용한다.
- 전표 유형은 표시용 구분에 가깝고, 기본 채번 시퀀스의 원천이 아니다.
- 부가세 자동라인은 부가세 계정코드 또는 매입/매출 세금계산서 영역이 사용될 때만 생성한다.
- 전자(세금)계산서 비고는 발행정보이며 신고·분개 금액에 반영하지 않는다.
- 복수거래는 최대 99개 상세를 한 전표 헤더에 연결하는 방식으로 취급하고, 분개/세무헤더 무결성을 함께 검증한다.
- 사업장은 기본 필수이고, 코스트센터/프로젝트/현장/사원 차원은 `DimensionConfig` 정책으로 제어한다.
- 테넌트 격리는 DB와 애플리케이션 계층 모두에서 강제한다. 스택이 글로벌 필터나 RLS를 지원하면 임의 `WHERE tenant_id`에만 의존하지 않는다.
- 회계 데이터는 이력 보존 지향이다. 회계 사실 데이터는 물리 삭제보다 취소, 역분개, 상태값, 이력 테이블로 처리한다.
- 기장 권한은 계약된 기장 모드와 주기장에 따라 달라진다. 비주기장의 마감, 결산, 신고 시도는 차단해야 한다.
- 민감정보는 기본 마스킹이며, 평문 허용 시 사유, 승인, Step-up, 감사기록이 필요하다.

## 설계-구현 체크리스트

코딩 전:

- 근거가 되는 설계 장과 구현할 정확한 규칙을 나열한다.
- 영향 산출물을 식별한다: UI, DB, API, 서비스, 배치, 연계, 알림, 보고서, 테스트.
- 동작을 바꿀 수 있는 미확정 정책 값이 23장에 있는지 확인한다.
- 새 추상화를 만들기 전에 기존 코드에서 가장 가까운 패턴을 찾는다.

코딩 중:

- 서버 경계에서 테넌트 컨텍스트, 권한 범위, 감사로그, 멱등성, 검증을 보존한다.
- 상태 머신은 UI뿐 아니라 트랜잭션 내부에서도 허용 전이를 강제한다.
- 분산/배치 워크플로우는 멱등 재시도와 오래된 락/상태 재검증을 포함한다.
- 보고서와 회계 산출물은 tie-out, 반올림, 기간 잠금, 감사 필요성을 검증한다.

코딩 후:

- 설계 규칙에 연결되는 테스트를 추가하거나 갱신한다.
- API 오류와 검증 메시지를 15장 기준으로 다시 확인한다.
- 테이블 제약과 상태값을 12장 기준으로 다시 확인한다.
- UI를 바꿨다면 0.2장의 화면 매핑과 동작을 다시 확인한다.

## 설계 변경 절차

먼저 변경 성격을 분류한다.

- 명확화: 동작 변경 없이 문구, 예시, 추적성만 보완한다.
- 확장: 새 옵션, 필드, 화면, API, 배치, 상태, 산출물을 추가한다.
- 파괴적 변경: 키, 워크플로우, 권한, 세무/회계 규칙, 공개 API 계약을 바꾼다.
- 정책 결정: 수치 기준, SLA, 요금, 법적 책임, 보존기간, 보안 승인, 기능 플래그 범위를 정한다.

변경은 일관되게 반영한다.

- 횡단 결정이나 화면 파생 결정은 0장 매트릭스를 갱신한다.
- 소유 도메인 장을 갱신한다.
- 데이터/API/배치/검증이 바뀌면 12-15장을 갱신한다.
- 보안, 개인정보, 관측성, 인프라, 컴플라이언스가 바뀌면 17-20장을 갱신한다.
- 기장 모드 책임이나 권한이 바뀌면 21장을 갱신한다.
- 결정이 미확정이면 23장에 남긴다.
- 연결 산출물을 만들어야 하거나 바꿔야 하면 24장을 갱신한다.

필요하면 기존 매트릭스 상태 용어인 `반영`, `보완`, `검토`, `연기`를 사용한다.

설계 변경이 다른 산출물로 전파되어야 하는 경우 `change-propagation.md`를 읽고 산출물별 영향 판정을 완료한다.
