# BK 설계 관리

## 운영 원칙

사용자가 다른 BK 설계 버전을 명시하지 않으면 `설계/bk_설계서_v*.md` 중 가장 높은 버전을 현재 기준 문서로 취급한다. 현재 기준은 `설계/bk_설계서_v1.0.md`다. 저장소 루트에서 `rg --files -g 'bk_설계서_v*.md' 설계`로 후보를 확인하고, 파일명의 `vMAJOR.MINOR` 숫자를 비교해 최신 버전을 고른다. 사용자가 특정 버전을 명시하면 그 버전을 우선한다. 한글 본문과 제목 검색에는 `rg -n`을 사용하고, PowerShell에서 문서 일부를 읽을 때는 `Get-Content -Encoding UTF8`을 사용한다.

코드나 설계서를 수정하기 전에 항상 추적성을 먼저 세운다.

- 적용되는 설계 장, 화면 ID, 엔티티, API, 상태값, 검증규칙, 미확정 정책을 식별한다.
- 23장의 정책 항목은 사용자나 기존 코드가 명확히 확정하지 않는 한 미확정으로 취급한다.
- 회계, 세무, 보안, 보존기간, 요금, 권한 규칙을 임의로 만들지 않는다. 필요한 가정은 좁게 명시한다.
- 구현과 설계가 충돌하면 충돌 내용을 먼저 알리고, 설계에 맞추거나 가장 작은 범위의 설계 변경을 명시한다.

## 작업 흐름

1. 기준 설계서와 관련 산출물의 위치를 찾는다. 기준 설계서는 명시 버전이 없으면 `설계/bk_설계서_v*.md`의 최신 버전이다.
2. `references/bk-design-map.md`에서 요청과 맞는 설계 장을 확인한다.
3. 설계 변경이면 `references/change-propagation.md`를 읽고 연관 산출물 전파 범위를 닫는다.
4. 설계 근거를 짧게 정리한다: 대상 장, 확정 규칙, 영향 모듈, 데이터/API/UI/배치 영향, 테스트, 미확정 결정.
5. 저장소의 기존 패턴을 따라 구현하거나 설계서를 수정한다.
6. 코드 동작과 설계 의무를 함께 검증한다.
7. 최종 응답에는 사용한 설계 장, 함께 갱신한 연관 설계, 남은 미확정 결정을 포함한다.

## 구현 지침

BK 설계서를 기준으로 코드를 변경할 때는 다음을 확인한다.

- 공통 관심사인 멀티테넌시, 권한, 감사/이력, 검증/오류, 알림, 관측성, 데이터 수명주기를 항상 점검한다.
- DB 변경에는 테넌트 격리, 키, 제약조건, 상태 필드, 이력/감사 기록, 마이그레이션/백필 방식을 포함한다.
- API 변경에는 권한 범위, 멱등성, 목록 페이징, 표준 오류, 감사로그, 민감정보 마스킹을 포함한다.
- UI 변경은 가능한 한 화면 ID 인벤토리에 매핑하고, 최신 최종 설계서의 전표 입력, 차원, 보조부, 기장 모드 결정을 보존한다.
- 배치나 워크플로우 변경은 멱등 재시도, 관측성, 락/상태 전이 재검증을 포함한다.
- 테스트는 수정한 코드 분기만이 아니라 설계 규칙 자체를 검증하도록 작성한다.

## 설계서 수정 지침

최신 `설계/bk_설계서_v*.md`를 수정할 때는 다음을 따른다.

- 기존 문서의 장 구성, 개정 이력 형식, 용어, 매트릭스 형식을 유지한다.
- 새 내용이 들어가는 문단만 고치지 말고 관련 위치를 함께 갱신한다.
- 횡단 결정은 0장의 해당 매트릭스와 영향을 받는 모듈 장을 함께 수정한다.
- 데이터, API, 배치, 검증, 산출물이 바뀌면 12장, 13장, 14장, 15장, 24장 중 해당 장을 갱신한다.
- 결정이 아직 미확정이면 확정 규칙처럼 쓰지 말고 23장에 남긴다.
- 사용자가 명시적으로 요청하지 않는 한 과거 버전 파일과 `설계/원본`, `설계/참조`의 원천 자료는 수정하거나 삭제하지 않는다.

## 설계 변경 전파

설계 문서가 바뀌면 같은 변경이 필요한 모든 관련 설계 산출물을 찾아 갱신한다. 변경 전파는 다음 조건을 모두 만족해야 완료로 본다.

- 원천 변경이 어느 장과 어느 산출물에서 시작됐는지 명시한다.
- 24장의 후속 산출물 목록을 기준으로 화면, DB, API, 배치, 연계, 보안, 운영, 테스트, AI/모델 설계 영향 여부를 판정한다.
- 영향이 있는 산출물은 실제로 수정하고, 영향이 없으면 "영향 없음" 근거를 남긴다.
- 기존 설계서 내부에서는 0장 매트릭스, 소유 도메인 장, 12-15장, 17-21장, 23-24장을 함께 확인한다.
- 화면 설계가 바뀌면 `설계/화면설계/bk_화면설계서_v*.html`의 최신 버전, `설계/화면설계/구현_화면_기초_v1.0.html`, `설계/화면설계/구현_화면_전표_v1.1.html`, `설계/원본/메뉴구조도_v4.0.html` 중 관련 파일을 확인한다.
- 기반 설계 규칙이 바뀌면 `설계/원본/bk_서비스_기본설계서.md`, `설계/원본/bk_세부설계_기초_v1.0.md`, `설계/참조`의 통합 전 원천 문서가 참조용으로 필요한지 확인한다.
- 변경 전파 절차와 산출물별 판단 기준은 `references/change-propagation.md`를 따른다.

## 응답 형식

규모가 있는 작업에서는 작업 결과나 중간 계획을 다음 구조로 정리한다.

- 설계 근거: 사용한 장 번호와 규칙.
- 변경 내용: 수정한 코드 또는 문서 산출물.
- 검증: 실행한 테스트, 확인, 남은 검증 공백.
- 미확정 결정: 아직 정책 또는 설계 결정이 필요한 항목.
- 전파 결과: 함께 갱신한 연관 설계와 영향 없음으로 판단한 산출물.

## 참고자료

요청을 올바른 설계 장으로 라우팅하거나, 구현 체크리스트를 만들거나, 설계서를 수정할 때는 `references/bk-design-map.md`를 읽는다.
설계 변경이 다른 설계 산출물에도 반영되어야 하는지 판단하고 빠짐없이 전파할 때는 `references/change-propagation.md`를 읽는다.
